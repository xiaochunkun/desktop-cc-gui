## Why

当前对话模板存在待修复问题，且问题将逐步补充。若不先建立统一提案，后续修复会缺少范围边界、验收口径与回归约束，容易出现修一个坏两个的问题。

## What Changes

- 建立“对话模板缺陷修复”统一变更入口，按 bug 逐条补充需求与任务。
- 为对话模板相关行为定义一致的修复规则：输入输出一致性、渲染稳定性、交互可预期性。
- 为每个已受理 bug 增加可验证的验收标准与最小回归检查。
- 不引入新业务功能，仅修复与稳定现有对话模板能力。

## 研究问题记录

### 问题 1：Codex 模式回复内容偶发重复

- 来源：用户实测截图（2026-02-27）
- 触发上下文：Codex 模式下进行简短问候类输入（示例：`你好`）
- 现象：助手文本在同一条回复中被重复拼接，表现为整句重复或段落重复（如“你好，我在。”出现两次）
- 期望：同一轮 assistant reply 在渲染层与消息层都只出现一次，禁止重复拼接
- 当前状态：已记录，待进入 design/specs/tasks 细化并落地修复
- 优先级：P0（影响基础可用性与用户信任）
- 影响范围：Codex 消息组装链路、流式片段合并逻辑、消息渲染层
- 复现步骤：
    1. 新建 Codex 会话
    2. 输入短文本（如 `你好`）
    3. 观察同一条 assistant 消息是否出现重复语句
- 验收焦点：同一 turn 在最终消息中只能有一份语义内容

### 问题 2：批量编辑文件列表缺少文件级点击弹出能力

- 来源：用户实测截图（2026-02-27）
- 触发上下文：Codex 模式执行“批量编辑文件”后，卡片内展示受影响文件列表
- 现象：文件条目仅展示文件名与增减行信息，无法点击进入弹出详情
- 参考：Git 模块已有类似“点击文件 -> 弹出/展开变更详情”的交互实现
- 期望：批量编辑卡片中的每个文件项支持点击，弹出该文件变更详情（交互与 Git 模块保持一致或高度一致）
- 当前状态：已记录，待在 design/specs 中明确复用策略与边界
- 优先级：P0（阻断核心审阅操作）
- 影响范围：批量编辑卡片组件、文件引用点击处理器、详情弹窗组件
- 复现步骤：
    1. 触发一轮包含多文件修改的批量编辑
    2. 在卡片中点击任一文件名
    3. 观察是否打开文件详情
- 验收焦点：所有文件项点击都能进入统一文件详情弹窗

### 问题 3：Plan 弹窗层级错误导致内容被遮挡

- 来源：用户实测截图（2026-02-27）
- 触发上下文：在批量编辑文件卡片点击右下角 `Plan` 按钮
- 现象：Plan 弹窗被其它 UI 层遮挡，显示层级错误，内容可见性受影响
- 期望：Plan 弹窗应始终位于当前上下文的最上层可视层级，不被卡片内容或邻近容器遮挡
- 当前状态：已记录，待在 design/specs 中明确 stacking context 与统一弹层策略
- 优先级：P0（功能可见但不可用）
- 影响范围：Plan 快览弹层、卡片容器层级、Portal 挂载策略
- 复现步骤：
    1. 打开含批量编辑卡片的消息
    2. 点击 `Plan`
    3. 检查弹层是否被遮挡、是否可完整交互
- 验收焦点：任意常见分辨率与滚动位置下，Plan 弹层完整可见可操作

### 问题 4：流式输出区域文件点击触发崩溃

- 来源：用户实测截图（2026-02-27）
- 触发上下文：在助手流式输出文本中点击“文件: xxx.rs”类型的文件链接
- 现象：点击后程序出现崩溃（未进入预期文件查看交互）
- 期望：该类文件链接点击后应打开文件详情弹窗（文件选择/预览弹窗），而不是触发崩溃
- 补充要求：交互行为与问题 2 中“批量编辑文件卡片”的文件点击弹窗能力保持一致
- 当前状态：已记录，待在 design/specs 中补充统一文件链接点击协议与异常兜底
- 优先级：P0（稳定性缺陷）
- 影响范围：流式文本渲染、文件链接解析、点击事件边界处理
- 复现步骤：
    1. 让助手输出包含文件引用的流式文本
    2. 点击文件引用
    3. 观察是否崩溃或失去交互
- 验收焦点：点击异常仅提示错误，不影响会话继续

### 问题 5：File changes 与批量编辑文件的信息职责重叠

- 来源：用户实测截图（2026-02-27）
- 触发上下文：同一轮输出中同时出现 `File changes` 与“批量编辑文件”卡片
- 现象：两处都在表达文件变更信息，用户感知为重复展示，认知负担偏高
- 期望：明确主从职责，避免同一语义在多个卡片重复铺陈
- 建议方向：`File changes` 作为唯一变更详情主视图；“批量编辑文件”保留摘要与操作入口（如 `Plan`）
- 当前状态：已记录，待在 design/specs 中细化去重规则与展示优先级
- 优先级：P1（信息架构与易用性）
- 影响范围：消息卡片编排层、operation 聚合展示逻辑
- 复现步骤：
    1. 触发一轮含 file diff 与批量编辑结果的输出
    2. 对比两张卡片是否重复展示同级详情
    3. 检查用户是否能在单一主视图完成详情审阅
- 验收焦点：同 operation 仅一个详情主视图，其他卡片不重复展开详情

### 问题 6：File changes 信息不完整且文件交互不统一

- 来源：用户实测截图（2026-02-27）
- 触发上下文：查看 `File changes` 卡片文件列表时
- 现象：
    - 缺少单文件级 `+n/-m` 增减统计
    - 缺少全部文件总计 `+total/-total` 统计
    - 缺少文件状态标识（新增/修改/删除）
    - 文件点击行为不统一，未稳定进入弹窗查看
- 期望：
    - 每个文件行展示状态标识（`A/M/D`）与对应增减统计
    - 卡片头部展示聚合增减统计（`+total/-total`）
    - 文件名点击后统一打开文件详情弹窗
- 当前状态：已记录，待在 design/specs 中明确字段来源、映射规则与 UI 展示规范
- 优先级：P1（可读性与操作一致性）
- 影响范围：File changes 卡片头部统计、文件行渲染、状态映射逻辑
- 复现步骤：
    1. 打开包含多个文件的 `File changes`
    2. 检查是否展示单文件 `+/-`、总计 `+/-`、`A/M/D`
    3. 点击文件名检查是否进入统一详情弹窗
- 验收焦点：统计准确且与详情一致，交互路径统一

## 缺陷分组与实施优先级

| 分组       | 包含问题    | 优先级   | 目标                 |
|----------|---------|-------|--------------------|
| 稳定性与可用性  | 1, 3, 4 | P0    | 消除重复回复、遮挡、崩溃，先保证可用 |
| 交互一致性    | 2, 4, 6 | P0/P1 | 所有文件引用点击行为统一为详情弹窗  |
| 信息架构与可读性 | 5, 6    | P1    | 降低重复信息与认知负担，补全统计语义 |

## 详细验收用例（补充）

1. 用例 A（回复去重）
    - GIVEN 新建 Codex 会话
    - WHEN 连续输入 5 轮简短问候/确认语
    - THEN 任一 assistant 消息不得出现整句重复
2. 用例 B（批量编辑文件可点击）
    - GIVEN 批量编辑卡片包含 N 个文件
    - WHEN 逐一点击文件项
    - THEN 每次都打开对应文件详情弹窗且内容匹配目标文件
3. 用例 C（Plan 弹层层级）
    - GIVEN 批量编辑卡片在可滚动消息列表中
    - WHEN 打开 `Plan`
    - THEN 弹层不被遮挡并可完成滚动、点击、关闭
4. 用例 D（流式文件点击稳定性）
    - GIVEN 流式输出含文件引用
    - WHEN 点击文件引用
    - THEN 应用不崩溃，失败场景仅出现可恢复提示
5. 用例 E（主从职责）
    - GIVEN 同 operation 同时存在 `File changes` 与批量编辑卡片
    - WHEN 查看详情
    - THEN 仅 `File changes` 展开详情；批量编辑卡仅保留摘要和操作
6. 用例 F（统计与状态）
    - GIVEN `File changes` 包含新增/修改/删除混合文件
    - WHEN 渲染卡片
    - THEN 文件行含 `A/M/D` 与 `+n/-m`，头部展示 `+total/-total` 且可由文件行求和验证

## 回归边界与不变式

- 不变式 1：不改变非 Codex 引擎（Claude/OpenCode）既有展示与交互行为。
- 不变式 2：不新增后端 API，不引入新依赖，不改变持久化数据结构。
- 不变式 3：所有文件点击入口统一进入同一详情交互协议，失败可恢复、不可崩溃。
- 不变式 4：同 operation 下，详情语义只出现一个主展示区。

## Capabilities

### New Capabilities

- `conversation-template-maintenance`: 规范对话模板缺陷修复流程、行为约束与回归门禁。

### Modified Capabilities

- `codex-chat-canvas-collaboration-mode`: 如 bug 涉及协作模式下模板展示/交互，将在后续 specs 中补充对应 requirement
  delta。
- `conversation-lifecycle-contract`: 如 bug 涉及会话状态与模板生命周期约束，将在后续 specs 中补充对应 requirement delta。

## Impact

- Affected code: 对话模板相关前端渲染与交互逻辑（以实际 bug 影响文件为准）。
- APIs: 无新增外部 API；仅可能调整内部状态流转与参数校验。
- Dependencies: 不新增依赖。
- Systems: 对话区 UI、模板状态管理、相关测试用例。

## 目标与边界

- 目标：在不改变现有产品定位的前提下，逐条修复你后续提交的对话模板 bug，并确保可回归验证。
- 边界：只处理“对话模板”相关缺陷，不扩展到无关模块与新功能开发。

## 非目标

- 不重做对话系统架构。
- 不引入新的模板编辑器或全新交互范式。
- 不做与 bug 无直接关联的样式大改。

## 技术方案对比

- 方案 A：分散修复（每个 bug 直接改代码，不建统一约束）
    - 优点：单点速度快。
    - 缺点：口径不一致，回归风险高，后续追踪成本高。
- 方案 B：统一提案 + 增量落地（本方案）
    - 优点：边界清晰、验收可追踪、可持续吸收新 bug。
    - 缺点：前置文档成本略高。
- 取舍：选择方案 B，以降低累计修复阶段的系统熵增和回归概率。

## 验收标准

- 每条 bug 在 specs/tasks 中都有对应条目与明确验收条件。
- 每次修复均可通过最小回归验证（至少覆盖受影响行为路径）。
- 修复后无新增破坏性变更（保持现有对话流程兼容）。
- 针对“问题 1”，在 Codex 模式下连续执行多轮简短输入测试时，任一单条 assistant reply 不得出现重复文本。
- 针对“问题 2”，批量编辑卡片中的文件项可点击打开变更详情，并正确显示对应文件的 diff 内容与行级增减信息。
- 针对“问题 3”，点击 `Plan` 后弹窗在桌面常见分辨率与滚动位置下均不被遮挡，且可完整交互。
- 针对“问题 4”，在流式输出区域点击任一文件链接时，应用不崩溃，并稳定打开目标文件详情弹窗。
- 针对“问题 5”，同一 operation 的文件变更详情只保留一个主展示区，其他卡片仅呈现摘要与操作入口，且不造成语义重复。
- 针对“问题 6”，`File changes` 必须同时展示单文件与总计增减统计、文件状态标识（`A/M/D`），且文件点击统一进入详情弹窗。
