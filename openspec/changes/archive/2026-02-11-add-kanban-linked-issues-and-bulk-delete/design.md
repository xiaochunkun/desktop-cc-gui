## Context

当前变更涉及两个前端交互面：

1) WorkspaceHome 右侧最近会话列表，需要从“单击进入”扩展到“可管理（多选/批量删除）”；
2) Composer 输入区上方，需要显式展示 Kanban 关联问题，减少上下文跳转成本。

现有代码已具备单条会话删除能力（removeThread + 草稿/图片清理）和 Kanban
关联面板数据源（linkedKanbanPanels），因此本次应以最小增量复用既有能力，避免重构底层线程与存储模型。

约束：

- 不引入新的后端 API；
- 不改变线程数据结构；
- 保持 TypeScript 类型与现有行为兼容；
- 高风险操作（批量删除）必须具备确认门禁。

## Goals / Non-Goals

**Goals:**

- 为最近会话提供“浏览态/管理态”双态交互，并支持批量删除。
- 在 Composer 输入区上方显式呈现 Kanban 关联问题入口。
- 复用现有删除链路，确保行为一致与低回归风险。

**Non-Goals:**

- 不做跨 workspace 批量删除。
- 不实现回收站、撤销删除、软删除改造。
- 不重做 Kanban 任务模型或线程存储协议。

## Decisions

### Decision 1: WorkspaceHome 采用双态交互（Browse / Manage）

- 选择：默认 Browse 态保持“点击进入会话”；进入 Manage 态后点击项改为“切换选中”，并显示批量操作栏。
- 原因：将“导航行为”和“管理行为”分离，可显著降低误删。
- 备选：在单一态上使用长按或快捷键触发多选。放弃原因：桌面端 discoverability 差，且行为不直观。

### Decision 2: 批量删除复用现有单删链路，不新增后端契约

- 选择：在 App 层新增批量删除回调，内部循环调用既有 removeThread，并同步清理对应草稿与附件缓存。
- 原因：复用既有稳定逻辑，降低代码路径分叉与一致性风险。
- 备选：新增后端批量删除 command。放弃原因：本次收益不足，额外增加接口与测试面。

### Decision 3: 删除安全策略采用“一次确认 + 条目数量提示”

- 选择：执行批量删除前显示确认弹窗，文案含将删除条目数量。
- 原因：防误操作是批量 destructive 操作的最低门槛。
- 备选：无确认直接删除。放弃原因：风险过高。

### Decision 4: Composer 顶部 Kanban 关联问题复用现有 linkedKanbanPanels 渲染位

- 选择：在输入区上方保持轻量 chip/strip 展示，可选中并跳转。
- 原因：已有数据与交互能力完整，属于胶水接入，最小改动可达成目标。
- 备选：新增独立弹层管理 Kanban 关联。放弃原因：复杂度高，不符合当前需求边界。

## Risks / Trade-offs

- [风险] 管理态切换后用户不易理解当前交互语义 → [缓解] 明确状态条（已选 N 条、全选、取消、删除）+ 按钮文案区分。
- [风险] 批量删除与会话列表刷新存在瞬时不同步 → [缓解] 删除期间禁用操作并串行执行，删除后统一刷新选择状态。
- [风险] Composer 顶部增加元素导致窄屏拥挤 → [缓解] 使用横向滚动与紧凑样式，保留移动端可用性。

## Migration Plan

1. 在 OpenSpec 中先固化 specs 与 tasks，明确验收边界。
2. 先改 WorkspaceHome（管理态/多选/批删），再接 App 回调。
3. 接入 Composer 顶部 Kanban 关联问题展示。
4. 补充 i18n 与测试，执行 lint/typecheck/test。
5. verify 通过后归档。

Rollback Strategy:

- 若出现回归，优先回退 WorkspaceHome 的 Manage 态入口与批删回调接入；
- 由于无数据结构变更，可文件级快速回退。

## Open Questions

- 批量删除确认文案是否需要显示具体标题列表（当前设计仅显示数量）。
- Composer 顶部关联条在 compact 模式下是否需要仅显示 icon（当前先保留文本+滚动）。
