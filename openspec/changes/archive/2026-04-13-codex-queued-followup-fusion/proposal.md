## Why

### 背景/问题

当前对话框上方的排队消息区域已经承担“运行中继续追问”的核心职责，但在体验上有两个明显断层：

1. 视觉断层：排队区域的外框、内层 item 和下方 composer 的圆角语义不统一，像是后贴上的列表，而不是同一个对话输入容器的一部分。
2. 交互断层：用户已经可以在运行中继续排队消息，但只能“继续等待”或“删除”，缺少类似官方 Codex 客户端 `Guide` 的单条升级入口，无法把某一条排队消息直接并入当前实时作答。

这不是协议能力缺失，而是现有能力没有被完整表达出来。当前实现已经具备：

- 排队中的 follow-up 消息容器；
- 运行中立即发送/steer 的基础能力；
- 线程内既有的消息发送、排队、删除和状态追踪链路。

因此本提案目标是做一次低风险收口：不改产品原有语义，不引入新模式，只把“排队”和“融合（Guide/Steer）”的边界表达清楚。

## 目标与边界

### 目标

- 目标 1：让排队区域在视觉上与 composer 整体圆角风格一致，消除割裂感。
- 目标 2：为每个排队项提供“融合”按钮，使其优先复用现有运行中 `Guide/Steer` 语义，把该条问题升级为当前作答上下文中的即时追问。
- 目标 3：保持现有队列、删除、发送和线程隔离语义不变，只新增更明确的升级入口。

### 边界

- 仅作用于对话输入区上方的排队消息区域及其 item 交互。
- 优先复用现有 queue/send/steer/interrupt 能力，不新增后端协议，不要求 Rust/Tauri 新命令。
- “融合”是对参考客户端 `Guide/Steer` 语义的产品化命名，不引入新的运行模式。
- 融合动作 MUST 优先复用当前运行中的 follow-up / steer 路径；仅当当前引擎或线程运行态不支持同轮追加但支持安全 cutover 时，才允许回退到显式 cutover。
- 不改变 `requestUserInput`、Plan 卡片、askuserquestion 等其它问答卡片能力。
- 不修改默认排队策略、快捷键主语义或多引擎整体交互模型。

## 非目标

- 不重做整个 composer 结构或消息幕布信息架构。
- 不新增“批量融合”或“多条合并发送”能力。
- 不改变排队项的 FIFO 基本顺序规则，除非用户明确点击某一项的“融合”按钮。
- 不把官方客户端的整套引导 UI 原样照搬，只对齐其单条 `Guide` 语义。

## 技术方案对比与取舍

| 方案 | 描述 | 优点 | 风险/成本 | 结论 |
|---|---|---|---|---|
| A | 仅修视觉样式，不新增按钮 | 风险最低，改动最小 | 只能解决“难看”，无法解决“不能融合” | 不采用 |
| B | 视觉统一 + 排队项级“融合”按钮，优先复用现有 steer/in-run follow-up 路径 | 收益直接、语义清晰、最贴近现有行为 | 需要补状态机、payload 保真与回归测试 | **采用** |
| C | 强制 interrupt 后再 immediate send | 表面上实现直接 | 容易把“并入当前作答”做成“切断后另起一轮”，语义跑偏 | 不采用 |
| D | 新增独立引导模式或后端专用融合协议 | 交互想象空间更大 | 偏离当前产品语义，回归面过大 | 不采用 |

取舍：采用 B。它严格收敛在用户提出的两项需求内，同时最大化复用已有队列与运行中追加能力。

## What Changes

### 具体改动方案

- 视觉统一：
  - 将排队区域升级为与 composer 同源的圆角容器语义，统一外层圆角、边框厚度、背景层级与上下间距。
  - 队列 item 改为“容器内子卡片”风格，避免出现外层圆角与内层直角/尖角冲突。
  - 当队列存在时，排队区域与下方输入框在视觉上形成一个连续整体，而不是两个彼此独立的盒子。
- 融合按钮：
  - 在每个排队项右侧新增“融合”按钮，并保留原有删除按钮。
  - “融合”按钮的产品语义对齐参考客户端的 `Guide/Steer Current Run`：将该条排队消息从等待队列提升为当前运行上下文中的即时 follow-up。
  - 按钮仅在当前线程和当前引擎处于可融合状态时可交互；不满足条件时应禁用或不展示，避免制造错误预期。
- 状态处理：
  - 点击“融合”后，系统应优先复用当前已有的 in-run follow-up / steer 路径，而不是默认执行 interrupt。
  - 当当前引擎或运行态不支持同轮 follow-up、但支持安全 cutover 时，系统才允许执行显式 cutover（例如 interrupt + immediate send）。
  - 被融合的排队项从队列中移除，其余排队项保持原相对顺序。
  - 若融合失败，系统必须恢复该条消息到原队列位置，并给出明确失败提示。
  - 融合必须保留原队列项的完整 payload，包括 `text`、`images` 与 `sendOptions`，不得默默降级为纯文本发送。
- 语义约束：
  - 不改变“排队”本身的含义；“融合”只是单条升级操作，不是新的默认策略。
  - 不改变当前线程的默认 collaboration mode 选择；融合仅作用于本次即时 follow-up。

### 交互说明

1. 当当前会话正在实时运行，用户继续输入新问题时，消息仍按现有行为进入排队区域。
2. 每条排队项右侧提供两个动作：`融合` 和 `删除`。
3. 用户点击 `融合` 后：
   - 系统优先尝试沿用当前运行中的 follow-up / steer 路径；
   - 仅当当前运行不支持同轮追加但支持安全 cutover 时，才执行显式切换；
   - 该排队项立刻作为当前线程的 follow-up 发送；
   - 该项从队列移除；
   - 剩余项继续保持排队。
4. 若当前运行已结束、当前引擎不支持该路径或线程处于不可升级状态，`融合` 不应伪装成可点击成功。
5. `删除` 的现有语义保持不变，不与 `融合` 冲突。

### 预期效果

- 队列区与对话框整体风格一致，视觉上更像完整输入容器的一部分。
- 用户无需等待队列自然轮转，就能对某一条高优先级追问执行“立即并入当前作答”。
- 参考客户端中 `Guide` 的能力，在 CodeMoss 中以“融合”的命名被更清楚地承接下来，同时不扭曲现有 steer 语义。
- 变更局限在前端表达层与既有发送编排层，不扩散到协议、存储或跨引擎语义。

## Capabilities

### New Capabilities

- `composer-queued-followup-fusion`: 定义排队消息区域的视觉统一与排队项级别“融合”动作，包括状态恢复、顺序保持与失败回滚语义。

### Modified Capabilities

- `codex-chat-canvas-collaboration-mode`: 补充 Codex 路径下排队项融合时对当前 collaboration payload 与默认 mode 稳定性的约束，确保该动作不会改写线程默认协作模式。

## 验收标准

- 排队区域外层圆角、边框和背景层级与下方 composer 保持一致，不再出现明显样式割裂。
- 每个排队项都可见 `融合` 按钮，并与 `删除` 按钮共存。
- 点击 `融合` 后，该条消息会脱离等待队列并立即参与当前实时对话作答。
- 在已支持 steer / in-run follow-up 的运行态下，融合 MUST 优先走同轮追加路径，而不是默认先 interrupt。
- 点击 `融合` 不会修改默认 collaboration mode，也不会改变其它排队项的相对顺序。
- 融合失败时，该条消息会恢复到原队列位置，且用户能看到失败反馈。
- 融合发送 MUST 保留原队列项的 `text`、`images` 与 `sendOptions`，不得丢附件、访问模式或其它逐条发送参数。
- 非支持路径下不会出现“按钮可点但无效果”的假交互。
- 现有排队、删除、自动轮转和线程隔离语义保持不变。
