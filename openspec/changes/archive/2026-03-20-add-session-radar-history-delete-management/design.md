## Context

当前 `workspace-session-radar-overview` 已定义“进行中 + 最近完成”的聚合与排序，但未定义“最近完成”的用户治理能力。随着最近完成记录累积，Radar 可读性下降，并且用户无法在产品内主动清理历史噪声。

本次变更需要在“设置 > 其他设置”新增“雷达历史管理”并落地连招删除链路（多选删除、部分失败保留、同页重试），同时保证：

- 不影响进行中会话聚合与执行链路。
- 删除结果在“设置管理视图”与“Radar 概览视图”之间一致。
- 删除期间防重入，避免并发请求导致脏状态。

## Goals / Non-Goals

**Goals:**

- 提供可发现的“雷达历史管理”入口与管理页。
- 支持单条删除、批量删除、部分失败保留与同页重试的连招流程。
- 删除后跨视图数据一致，不出现“删后回弹”。
- 保持现有 Radar 进行中会话能力完全兼容。

**Non-Goals:**

- 不实现自动清理策略（TTL 定时删除、按天自动归档）。
- 不删除真实会话实体，仅删除 Radar 最近完成历史索引。
- 不改动后端会话执行协议与 thread 生命周期语义。

## Decisions

### Decision 1: 在设置域承载删除能力，而不是 Radar 主面板内联操作

- 选择：将删除入口放在“设置 > 其他设置 > 雷达历史管理”。
- 理由：管理行为与阅读行为解耦，避免 Radar 主面板因确认态/失败态交互变得拥挤。
- 备选：直接在 Radar 面板内删除。短期更快，但信息架构混杂，后续扩展成本高。

### Decision 2: 历史删除采用“结果驱动同步”策略

- 选择：删除请求返回后，以后端结果为准更新本地 `recentCompleted` 索引，再通知视图刷新。
- 理由：避免 optimistic update 在部分失败场景产生“删后回弹”。
- 备选：先前端移除再回滚失败项。交互更快，但状态机复杂且回滚闪烁风险高。

### Decision 3: 连招状态机固定四态

- 选择：`idle -> confirming -> deleting -> settled`，其中 `settled` 可继续进入下一轮删除。
- 理由：确保同页可连续执行多轮删除，且删除期间天然防重入。
- 备选：每次删除后强制退出管理态。实现简单，但破坏“连招”效率。

### Decision 4: 部分失败保留失败项选中态

- 选择：批量删除部分失败时，仅移除成功项；失败项及其选中态保留，支持一键重试。
- 理由：减少用户重复选择成本，符合管理动作的高频重试预期。
- 备选：全部清空选中。会导致失败重试成本高，违背“连招”目标。

## Risks / Trade-offs

- [删除与 Radar feed 异步刷新导致短暂不一致] → 删除成功后统一触发 feed 重算，且按结果集差量更新。
- [并发点击导致重复请求] → `deleting` 态禁用触发控件，并在 handler 层做重复提交守卫。
- [部分失败错误信息不可读] → 统一错误分类与摘要结构，保留失败项可重试。
- [新增设置入口增加认知路径] → 在“其他设置”下明确“雷达历史管理”命名，并保持入口单层可达。

## Migration Plan

1. 定义历史删除 command/hook 的输入输出契约（单条/批量统一返回结构）。
2. 在设置页新增“雷达历史管理”入口与管理页骨架。
3. 实现删除状态机与连招交互（确认、执行、防重入、失败保留、重试）。
4. 将删除结果同步到 Session Radar feed 源，保证跨视图一致。
5. 补齐单测与回归验证（部分失败、二次删除、删后一致性）。

回滚策略：

- 通过 feature flag 关闭“雷达历史管理”入口与删除动作，保留现有 Radar 只读能力。
- 删除逻辑不改动进行中会话链路，回滚不影响核心会话执行。
