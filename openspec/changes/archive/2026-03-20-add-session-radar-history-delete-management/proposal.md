## Why

Session Radar 的“最近完成”会话会持续累积，当前缺少用户可控的清理入口，导致列表噪音上升、定位成本变高。现在需要在“其他设置”中新增“雷达历史管理”删除能力，让用户可主动清理历史并保持 Radar 可读性。

## 目标与边界

- 目标：在“设置 > 其他设置”提供“雷达历史管理”，支持删除 Session Radar 的最近完成历史记录。
- 边界：本提案仅覆盖 Radar 历史管理，不改变进行中会话聚合、不改变会话详情内容、不改变跨 workspace 的运行态行为。

## 非目标

- 不引入“自动定时清理”策略。
- 不改动会话实体本身（删除 Radar 历史不等于删除真实会话）。
- 不在本提案中扩展为通用数据治理中心（仅针对 Radar 历史）。

## What Changes

- 在“其他设置”新增“雷达历史管理”入口与管理视图。
- 提供删除动作（单条删除 + 批量删除）用于清理“最近完成”历史。
- 删除流程包含确认态、执行态、防重入与结果反馈（全成功/部分失败/失败）。
- 清理后 Radar 概览与设置页数据保持一致刷新，避免残留脏数据。
- 增加“连招操作链路”约束：支持用户在同一管理会话中连续执行“筛选/多选 → 删除 → 查看失败项 → 重试删除”。

## 连招场景（补充）

- 连招 1：进入“雷达历史管理”后，用户可连续删除多批历史项，不需要反复退出设置页。
- 连招 2：首次批量删除部分失败后，失败项保留选中，用户可直接二次确认重试。
- 连招 3：删除执行期间禁止重复提交；执行完成后立即恢复可操作态并刷新最新列表。

## 技术方案对比

### 方案 A：仅在 Session Radar 面板内做内联删除

- 优点：开发路径短，交互链路最少。
- 缺点：管理操作与“设置治理”语义不一致；批量操作与确认流程会挤占 Radar 主阅读区域。

### 方案 B：在“其他设置”提供独立“雷达历史管理”（本次选择）

- 优点：信息架构清晰，符合“管理动作在设置域”原则；更容易承载批量操作、失败重试和后续策略扩展。
- 缺点：需要新增设置页入口与数据同步链路，初始改动面更大。

取舍：选择方案 B，优先保证长期可维护性与信息架构稳定性。

## Capabilities

### New Capabilities

- 无

### Modified Capabilities

- `workspace-session-radar-overview`: 扩展“最近完成”会话的生命周期管理，新增可删除与删除后一致性刷新要求。

## 验收标准

- 用户可在“设置 > 其他设置”进入“雷达历史管理”。
- 用户可删除单条最近完成记录，并在成功后从列表中移除。
- 用户可批量删除多条记录，且删除期间删除按钮防重入。
- 批量删除在部分失败时，仅移除成功项，失败项保留并可重试，页面展示失败摘要。
- 删除完成后，Session Radar 主视图与设置管理视图数据一致（无幽灵记录、无重复回填）。
- 用户在一个管理会话内可完成至少两轮连续批量删除，且交互状态机保持稳定（无卡死、无重复请求）。

## 风险与回滚

- 风险：删除接口与 Radar 聚合刷新时序不一致，可能造成短暂“删后回弹”。
- 控制：删除成功后以存储结果为准刷新来源数据，再触发 UI 重算；失败项仅本地保留，不反写成功项。
- 回滚：通过 feature flag 关闭“雷达历史管理”入口，回退到只读 Radar 视图，不影响进行中会话链路。
