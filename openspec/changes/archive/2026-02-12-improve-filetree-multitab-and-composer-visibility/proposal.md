## Why

当前文件树交互存在两个连续体验断点：

1. 右侧文件树一次只能打开 1 个文件，点击新文件会替换掉当前文件，无法并行对比。
2. 打开文件查看后，主页面 AI 输入框不可见，导致“看代码 -> 指定问题行 -> 发给 AI”链路中断。

该问题会直接拉低排查效率，尤其在跨文件定位 bug 场景下影响明显。

## What Changes

- 在右侧文件查看区引入多文件 Tab 能力（打开、切换、关闭、活动态保持）。
- 将主页面 Composer 调整为文件查看态下仍可见、可输入、可发送。
- 保持现有线程/消息发送链路不变，仅改造前端 UI 交互与装配层状态。
- **无 BREAKING 变更**：不修改消息协议、不调整后端存储模型。

## 目标与边界

- 目标
    - 支持在文件树中同时打开多个文件并快速切换。
    - 在文件查看过程中始终保留 AI 输入能力。
    - 消除“查看文件”和“提问 AI”互斥关系。
- 边界
    - 仅覆盖文件浏览区和主页面输入区布局/交互。
    - 仅作用于当前 workspace 的会话 UI 状态。
    - 不引入新的后端 command。

## 非目标

- 不做 IDE 级多分栏拖拽布局。
- 不新增复杂代码编辑能力（如 LSP、断点、重构面板）。
- 不做跨 workspace 的文件标签恢复与同步。
- 不改造既有 AI 引擎请求参数与会话持久化协议。

## 技术方案对比

### 方案 A：保留单文件视图，仅增加“固定当前文件（pin）”

- 做法：继续单视图，用户通过 pin/unpin 手动保留文件。
- 优点：改动小。
- 缺点：无法真正并行查看多个文件，切换成本依旧高；不满足核心诉求。

### 方案 B：引入文件 Tab 列表 + Composer 常驻（推荐）

- 做法：维护 `openFileTabs + activeFileTabId`，文件点击行为改为“打开/激活 Tab”；Composer 从互斥区改为固定可见区。
- 优点：直接解决两个核心痛点，路径清晰，回归可控。
- 缺点：需要处理 Tab 生命周期与布局滚动边界。

### 取舍

采用 **方案 B**。本次需求的关键是消除互斥视图，方案 B 在最小业务入侵前提下可完整达成目标。

## Capabilities

### New Capabilities

- `filetree-multitab-open`: 文件树支持多文件并行打开与标签化切换。
- `composer-persistent-input-with-filetree`: 文件查看态下，Composer 保持可见与可用。

### Modified Capabilities

- （无）当前主 specs 未定义文件树多 Tab 与输入框常驻能力，采用新增能力建模。

## Impact

- Affected UI surface
    - 右侧文件树与文件查看区。
    - 主页面底部 Composer 区域。
- Affected code areas（目标代码库）
    - `src/App.tsx`（状态装配）
    - `src/features/files/**`（文件树点击与文件查看）
    - `src/features/composer/**`（输入区容器可见性）
    - `src/styles/**`（布局与滚动样式）
- Dependencies
    - 复用现有文件打开与线程发送逻辑，不新增第三方依赖。
- Risk
    - 主要风险为布局遮挡、滚动冲突和 Tab 状态同步错误；通过状态机约束与回归测试控制。

## 验收标准

- 点击第二个文件时，第一个文件仍保留在 Tab 列表中，不被关闭。
- 可在 Tab 间切换并关闭任意 Tab；关闭活动 Tab 后焦点行为可预测。
- 文件查看区展开时，Composer 始终可见且可输入、可发送。
- 从文件内容切换到发送消息过程中，输入草稿不丢失。
- 现有线程切换与消息发送主流程无回归。
