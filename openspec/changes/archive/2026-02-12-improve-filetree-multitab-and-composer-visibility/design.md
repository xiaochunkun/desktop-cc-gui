## Context

当前 UI 把“文件查看”和“AI 输入”做成了互斥关系：

- 文件树点击文件后，查看区使用单文件模型，后续点击会覆盖先前文件。
- 文件查看态会挤占主内容高度，Composer 失去可见性。

本次设计目标是在不改后端契约的前提下，让文件定位与 AI 沟通可并行进行。

约束：

- 不新增后端 API/Command；
- 不改变消息发送与线程存储模型；
- 维持现有快捷键与输入行为兼容；
- 改动优先集中在装配层和文件视图模块。

## Goals / Non-Goals

**Goals:**

- 引入文件多 Tab 状态模型，支持打开/切换/关闭。
- 确保任何文件查看状态下，Composer 都可见、可输入、可发送。
- 降低跨文件排查时的上下文丢失。

**Non-Goals:**

- 不实现多分栏拖拽布局。
- 不实现跨 workspace 的 Tab 持久化。
- 不重构整个页面为 IDE 架构。

## Decisions

### Decision 1: 文件查看状态模型采用 `openFileTabs + activeFileTabId`

- 选择：统一维护 Tab 数组与活动 Tab id；点击文件时若已打开则激活，未打开则追加。
- 原因：状态明确、可测试、可预测。
- 备选：仅保存“最近两个文件”临时切换。放弃原因：扩展性弱，行为不稳定。

### Decision 2: 关闭 Tab 行为采用“就近回退”

- 选择：关闭活动 Tab 后优先激活右侧相邻 Tab，不存在则激活左侧。
- 原因：符合常见编辑器心智模型，减少用户迷失。
- 备选：每次关闭都回到首个 Tab。放弃原因：跳转突兀。

### Decision 3: Composer 采用底部常驻容器，而非条件渲染

- 选择：Composer 与文件查看区并存，保持固定可见高度；文件区独立滚动。
- 原因：直接消除“看文件时不能提问”的核心断点。
- 备选：切回浮层输入框。放弃原因：焦点/键盘行为复杂，且遮挡代码内容。

### Decision 4: 保持现有消息发送链路不变

- 选择：仅调整 UI 容器布局，不改发送逻辑、草稿逻辑与线程生命周期。
- 原因：降低回归风险，符合最小改动原则。
- 备选：重构输入链路与文件引用协议。放弃原因：超出本次范围。

## Risks / Trade-offs

- [风险] 双滚动容器导致体验割裂。
    - [缓解] 页面主容器禁双重滚动，文件区独立滚动，Composer 区域固定。
- [风险] Tab 状态与当前文件内容不同步。
    - [缓解] 用唯一 fileKey 驱动渲染，禁止 index 作为主键。
- [风险] 小窗口下可用空间不足。
    - [缓解] 设置最小高度与紧凑模式，优先保证输入框可见。

## Migration Plan

1. 新增 Tab 状态模型与行为函数（open/activate/close）。
2. 接入文件查看区 Tab UI（标签栏 + 内容区）。
3. 改造主页面布局，固定 Composer 可见。
4. 补充关键测试：多 Tab 行为、Composer 可见性、发送无回归。
5. 验证通过后进入 OpenSpec verify/归档流程。

Rollback Strategy:

- 如出现回归，优先回退 Tab 接入层与布局变更文件；
- 由于无后端与数据结构修改，回滚为纯前端文件级回退。

## Open Questions

2. 是否在后续迭代增加“从文件行号一键插入引用到输入框”的增强能力？
