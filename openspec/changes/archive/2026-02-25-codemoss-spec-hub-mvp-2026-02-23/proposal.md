## Why

CodeMoss 目前已经在项目层使用 OpenSpec，但产品交互仍以“文件浏览 + 手动命令”为主，规范、实现与验证没有形成稳定闭环。现在启动
Spec Hub MVP，是为了把规范驱动开发从“专家习惯”升级为“默认工作流”，降低团队协作摩擦并提升变更可追踪性。与此同时，当前仍有大量老项目和新项目在首次打开时没有
OpenSpec 目录，用户无法在 Spec Hub 内完成“从 0 到 1”的初始化和项目背景建立，这会导致后续规范更新不可持续。
此外，真实场景中还存在“代码仓库在 A 盘，OpenSpec 规范在 B 盘”的分离式部署；如果 Spec Hub 只能扫描工作区内 `openspec/`
，将导致这类项目被误判为未接入规范。

## 目标与边界

- 目标：
    - 在客户端提供统一的 Spec Hub 入口，覆盖 change 浏览、artifact 查看、关键动作执行与验证反馈。
    - 先把 OpenSpec 路径做深做稳，再为 spec-kit 提供最小兼容入口，避免一次性做过宽。
    - 让 change 状态由系统计算与展示，而不是依赖人工口头同步。
- 边界：
    - 本提案只覆盖规范工作流集成，不包含完整代码编辑器替代方案。
    - 本提案首期不承诺 spec-kit 的全量执行语义，只提供 detect/read-only/外部跳转。

## 非目标

- 不在本期引入新的 AI 引擎协议或替换现有对话协议。
- 不在本期重构所有历史 spec，仅对新 change 工作流生效。
- 不在本期实现跨仓库聚合治理（multi-repo federation）。

## What Changes

- 新增 Spec Hub 工作台信息架构：
    - Changes 列表（按状态分组 + 筛选）
    - Artifacts 视图（proposal/design/spec/tasks/verification）
    - 右侧执行台（动作/门禁/时间线/环境诊断 Tab）
    - 在执行台中展示执行结果与验证反馈（结构化失败定位）
- 新增统一 Spec Runtime：
    - 统一 change/artifact/task 状态模型
    - 由文件系统与 CLI 结果重建当前状态
- 新增 OpenSpec Adapter（首要支持）：
    - 命令执行、结果解析、前置条件检查、错误结构化
- 新增 spec-kit 最小兼容 Hook（非完整 Adapter）：
    - workspace 识别
    - read-only 信息映射
    - 外部命令/文档跳转入口
- 新增 Environment Doctor：
    - Managed/BYO 双模式
    - 依赖探测、版本诊断、失败恢复路径
- 新增 OpenSpec 初始化引导（无规范工作区场景）：
    - 当 workspace 未检测到 OpenSpec 结构时，提供「初始化 OpenSpec」引导入口
    - 区分老项目接入与新项目接入两条路径，避免单一模板误导
- 新增项目信息收集与后续更新能力：
    - 初始化过程中收集项目背景（领域、架构、约束、关键命令、负责人等）
    - 支持后续增量更新项目信息，确保规范上下文可持续演进
- 新增自定义 Spec 位置配置能力（分离式部署）：
    - 支持在 Spec Hub 中配置项目级 Spec 根目录（默认：`<workspace>/openspec`，可切换为外部绝对路径）
    - 配置后，provider 检测、change 列表、artifact 读取、执行动作与验证均基于该路径解析
    - 对不可访问路径、无效路径、权限不足路径提供结构化错误与一键恢复默认路径
- 新增 Tasks 清单交互写回能力（任务状态显式维护）：
    - Tasks Tab 中的任务复选框支持直接点击，系统将对应状态写回当前 `tasks.md`
    - 复选框写回成功后，任务进度统计与动作可用性（尤其归档门禁）需同步刷新
    - `验证` 动作仅负责规范校验，不自动修改任务复选框，避免“机器校验替代人工确认”

## 技术方案对比与取舍

| 方案                                            | 描述                           | 优点               | 风险                   |
|-----------------------------------------------|------------------------------|------------------|----------------------|
| A. 纯 CLI 模式                                   | 保持现状，仅在文档中补流程说明              | 实现成本最低           | 体验割裂，落地率低，状态不可视      |
| B. 直接做全量双适配                                   | OpenSpec + spec-kit 同时完整执行支持 | 功能覆盖最大           | 语义差异大，首期风险高，交付周期长    |
| C. OpenSpec-first + spec-kit minimal hook（采用） | OpenSpec 全链路，spec-kit 最小兼容入口 | 价值闭环快、风险可控、可渐进扩展 | 需要清晰标注 spec-kit 支持边界 |

取舍结论：采用 C，优先保证“可用且稳定”的主路径，再根据使用数据决定是否升级 spec-kit 支持级别。

- `spec-hub-runtime-state`: 增加任务写回后的状态重建与门禁同步规则。

## 验收标准

1. 在 OpenSpec workspace 下，用户可在 Spec Hub 完成一次 `change -> apply -> verify` 的闭环操作，且状态实时刷新。
2. 当缺少前置 artifact（如 `tasks.md`）时，执行台动作 Tab 必须给出明确 blocker，并阻止错误执行。
3. 环境缺失（如未找到 `openspec`）时，Doctor 能给出可执行修复建议，且系统可降级为只读。
4. 在 spec-kit workspace 下，系统能正确识别为 minimal 支持并提供可用的 read-only + 外部跳转入口。
5. 验证失败结果必须包含可定位信息（失败动作、change 名称、失败原因），并可在执行台直接查看。
6. 当用户打开一个不含 OpenSpec 结构的老项目时，Spec Hub 必须提供初始化引导，并支持在引导中预填/确认项目背景信息。
7. 当用户打开一个新项目且尚未建立 OpenSpec 规范时，Spec Hub 必须提供最小可用初始化路径，确保初始化完成后可直接进入规范工作流。
8. 项目信息更新后，系统必须保留可追踪记录，并在后续规范变更中可被读取复用。
9. 当项目工作区内不存在 `openspec/` 但用户配置了外部 Spec 根目录时，Spec Hub 必须正确识别 provider、加载 changes 与
   artifacts。
10. 当外部 Spec 根目录无效（不存在/无权限/结构不完整）时，系统必须阻断执行动作并展示可操作修复提示（重新选择路径或恢复默认）。
11. 当用户在 Tasks Tab 点击任务复选框时，系统必须将状态写回 `tasks.md`；写回失败时必须回滚 UI 勾选并提示失败原因。
12. `验证` 通过后不得自动勾选任何任务；`归档` 前必须满足“验证通过 + 必需任务完成”门禁。

- Tooling：引入环境探测和模式管理（Managed/BYO）逻辑。
- Delivery：未来与规范相关的功能迭代将通过 Spec Hub 能力进行扩展，而非零散拼接。
- Onboarding：新增无 OpenSpec 场景下的初始化与项目背景维护闭环，降低旧项目接入门槛。

- Workflow 可信度：任务复选框由人工显式确认并写回规范文件，避免“已验证但任务文档未对齐”的漂移。

## 追加提案：会话级 Spec 关联与上下文注入（2026-02-24）

### Why

当前虽然已支持外置 Spec 根目录配置，但新开会话后，AI 侧的“规范文件读取协议”仍可能只按默认路径与项目名推导路径扫描，导致：

- Spec Hub 可见 `resolvedSpecRoot`，但会话内助手回答“看不到你的 spec”；
- 用户在同一 workspace 下重复配置与重复解释上下文；
- 规范驱动流程在“新会话入口”断裂，形成错误认知。

这属于“UI 层已关联、会话层未关联”的一致性缺口，必须补齐。

### What Changes

- 新增 `session-spec-linking` 能力，将 workspace 的 `resolvedSpecRoot` 注入新会话上下文：
    - 会话创建时读取当前 workspace 的 spec root（默认或自定义）；
    - 将该路径加入规范扫描白名单，而非仅依赖项目名推导规则；
    - 在会话启动日志中显式回显“已关联 spec 路径”。
- 新增会话级 spec 可见性探测与提示：
    - 若路径可读且结构有效，返回“已关联并可读取”；
    - 若路径失效（不存在/无权限/结构缺失），返回结构化修复建议；
    - 提供“一键重新绑定到 Spec Hub 当前路径”入口。
- 新增一致性约束：
    - Spec Hub 的 `resolvedSpecRoot` 变更后，新会话默认继承；
    - 若继承失败，必须在会话首屏显示阻断提示，避免 silent fallback。

### 验收标准（追加）

13. 当 workspace 已配置外置 Spec 根目录（如 `/Users/chenxiangning/code/AI/github/codemoss-openspec/openspec`）时，新建会话后
    AI 必须可读取该路径下规范文件。
14. 在新会话中询问“你能看到我的 spec 吗”，系统必须基于真实探测结果返回，不得回退为仅项目名推导。
15. 当 spec root 无效时，会话首屏必须给出结构化错误与修复动作（重新绑定/恢复默认）。
16. Spec Hub 修改 spec root 后，新会话无需手工重复配置即可继承最新关联。

### Impact（追加）

- Session Runtime：新增会话启动期 spec 路径注入与探测流程。
- UX 一致性：消除“Spec Hub 看得到 / 会话看不到”的割裂行为。
- 规范可信度：确保“已配置 = 可读取 = 可回答”，减少错误答复。

## 追加提案：归档失败 AI 接管修复（2026-02-24）

### Why

当前链路中，`validate --strict` 与 `archive` 的语义边界不同：前者校验 change 包，后者执行 delta 与主 spec 的合并。
在真实使用中会出现“验证通过但归档失败（header not found / target spec mismatch）”的情况，用户只能手工分析日志并反复修改，流程中断严重。

### What Changes

- 在执行台动作区新增“归档阻塞 AI 接管”入口：
    - 当 archive 存在 blocker，或时间线存在最近一次 archive 失败事件时，提供一键接管修复。
- 接管执行上下文自动收集并传递给智能体：
    - 当前 changeId
    - 当前 resolved spec root（默认或外置路径）
    - 当前 archive blockers
    - 最近一次 archive 失败输出
- 接管执行方式：
    - 通过 engine sync 通道调用所选智能体（codex/claude/opencode），以 full-access 模式执行修复。
    - 接管目标限定为“修复 delta 可归档性 + 重新验证”，不默认自动归档。
- 执行后自动刷新 Spec Hub 状态，并在动作区回显 AI 接管结果，便于复核与下一步归档。

### 验收标准（追加）

17. 当 archive action 被 blocker 阻断时，动作区必须出现“AI 接管修复”入口，并展示阻断明细。
18. 当 archive 命令失败（包括 exitCode=0 但语义失败）后，动作区必须可基于最近失败输出触发 AI 接管。
19. AI 接管执行后，Spec Hub 必须自动刷新，并在动作区展示接管结果文本。
20. 在配置外置 spec root 的 workspace 中，AI 接管必须沿用当前 resolvedSpecRoot，不得回退到默认路径。

### Impact（追加）

- 归档可用性：将“人工排障”转为“可接管修复”，缩短 blocked change 的恢复时间。
- 交互闭环：把失败上下文直接注入处理链路，减少用户重复复制日志。
- 风险控制：保持“AI 修复 + 用户复核 + 再归档”的可控流程，不引入 silent auto-archive。

## 追加提案：AI 接管执行可视化（2026-02-24）

### 背景

当前“AI 接管”入口已经可触发，但用户在点击后缺少可观测执行反馈：无法判断是否已开始、是否在进行、卡在哪个阶段、是成功还是失败。这会直接影响用户对“按钮是否可用”的信任，也会导致重复点击与误判。

### 目标

- 在 Spec Hub 内提供 AI 接管执行过程的实时可见性（running/progress/result）。
- 在不离开当前页面的前提下，让用户明确知道：
    - 任务是否真的开始执行；
    - 当前处于哪个阶段；
    - 失败时具体报错是什么；
    - 成功后是否已触发状态刷新。

### What Changes

- 在“归档阻塞 AI 接管”卡片中新增执行状态区：
    - `准备中` / `执行中` / `成功` / `失败` 状态标签；
    - 最近一次运行时间与耗时；
    - 分阶段进度（已发起、等待智能体响应、解析结果、刷新 runtime）。
- 新增可滚动运行日志视图：
    - 按时间追加关键信息行；
    - 保留最近一次成功/失败摘要；
    - 失败时展示可复制的错误详情（命令/关键信号/建议动作）。
- 统一按钮交互语义：
    - 执行中按钮进入 loading + 禁止重复触发；
    - 执行结束后恢复可点击；
    - 明确展示“本次是否完成自动刷新”结果。

### 验收标准（追加）

21. 点击“AI 接管”后 300ms 内必须看到明确的进行中状态（而非静默等待）。
22. 执行过程中，用户必须能看到至少 3 个阶段节点（发起、执行、收尾）中的当前阶段与已完成阶段。
23. 执行失败时，必须展示结构化错误摘要（失败阶段 + 失败原因 + 建议下一步）。
24. 执行成功后，必须展示“已刷新/刷新失败”结果，避免用户误判状态。

### Impact（追加）

- 可观测性：将“黑盒按钮”升级为“透明执行流”。
- 可用性：降低重复点击与误操作。
- 可诊断性：失败定位成本下降，便于后续自动化修复迭代。

## 追加提案：继续/执行结果可视化与一键闭环（2026-02-24）

### 背景

当前 `继续`（`openspec instructions specs`）与 `执行`（`openspec instructions tasks`
）虽然可触发命令，但用户点击后常感知不到“发生了什么、产出了什么、下一步该做什么”，形成“按钮可点但价值不可见”的体验。
这会导致两个直接后果：

- 用户认为按钮无效或不稳定，重复点击与误判增加；
- 客户端路径价值低于“直接找 AI 对话执行”，Spec Hub 的流程入口意义被削弱。

### 目标

- 把 `继续/执行` 从“命令触发按钮”升级为“结果驱动入口”；
- 用户在一次点击后必须获得：执行状态、产出摘要、下一步动作；
- 当本次无有效产出时，明确告知“无新增建议”，避免静默。

### What Changes

- 为 `继续/执行` 增加统一执行反馈卡片（Action Result Card）：
    - 点击后 300ms 内显示运行态（`准备中/执行中/成功/失败`）；
    - 展示开始时间、耗时、最近一次执行结果。
- 为 `继续/执行` 增加结构化结果摘要（替代纯日志感知）：
    - `本次产出`：新增/修正建议项、涉及 artifact；
    - `无产出`：明确说明“当前无新增建议”；
    - `下一步`：给出推荐动作（执行/验证/归档）并提供一键跳转/触发。
- 为失败场景提供显式恢复链路：
    - 重试当前动作；
    - 查看原始输出；
    - 命中可修复语义失败时，提供 AI 接管入口。
- 为动作链路增加“结果一致性文案约束”：
    - 所有动作结束后必须回显“动作 + 结果 + 下一步”；
    - 禁止仅写入时间线而无当前视图反馈。

### 验收标准（追加）

25. 点击 `继续` 或 `执行` 后，300ms 内必须出现进行中状态，不允许静默等待。
26. `继续` 成功后必须在当前面板展示“本次 specs 指导摘要”，且可识别“有新增建议/无新增建议”。
27. `执行` 成功后必须在当前面板展示“本次 tasks 指导摘要”，且给出明确下一步（通常为验证或继续执行）。
28. 任一动作失败时，面板必须展示结构化失败信息（失败阶段、失败原因、建议动作），并保留查看原始输出入口。
29. 当动作结果判定为“无新增建议”时，必须显式显示无产出提示，不得仅更新时间线。
30. `继续/执行` 结果必须与时间线记录一致（动作类型、时间、成功失败状态一致）。
31. 在外置 spec root 场景下，`继续/执行` 的结果解析必须沿用当前 `resolvedSpecRoot`，不得回退默认路径。

### Impact（追加）

- 交互价值：把“能点”升级为“可理解、可决策、可推进”。
- 流程闭环：用户无需切到聊天窗口才能知道下一步。
- 一致性：动作反馈、时间线、门禁状态三者形成同源语义。
