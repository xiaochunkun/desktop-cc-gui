## Context

Codemoss 当前在高分辨率显示器场景下，存在文本和控件视觉尺寸偏小、主幕布有效阅读宽度不足的问题。该问题跨越设置模块、主布局模块、以及 Tauri 窗口桥接能力，属于跨层改动。  
技术约束：
- 现有架构为 React + Tauri IPC，缩放能力应优先复用 Tauri window/webview 能力。
- 需要保持现有业务功能与数据模型不变，避免引入破坏性变更。
- 需要支持设置持久化并在应用重启后恢复。

相关方：
- 终端用户：需要在高 DPI 下获得可读可用界面。
- 前端研发：需要稳定、低维护成本的缩放实现。
- 桌面端集成：需要窗口缩放能力与配置系统安全协作。

## Goals / Non-Goals

**Goals:**
- 提供全局 UI 缩放设置（80%~150%）并即时生效。
- 将缩放值持久化并在启动时恢复。
- 在主要区域（侧栏、主幕布、右侧面板、设置页）保持布局稳定和可操作性。
- 支持“一键恢复 100% 默认值”。

**Non-Goals:**
- 不提供按模块独立缩放。
- 不改造主题系统或字体体系。
- 不替代操作系统级缩放能力。

## Decisions

### 决策 1：采用 WebView/Window 级全局 zoom factor 作为主实现路径

- 方案 A（采用）：Tauri 侧提供统一缩放桥接命令，前端设置更新后直接触发 zoom factor 变更。
- 方案 B（未采用）：仅靠 CSS `font-size`/`transform: scale` 做模拟缩放。

选择理由：
- A 与“全局缩放”语义一致，覆盖面完整，避免组件级碎片化适配。
- B 在弹层定位、点击热区、滚动容器方面风险更高，长期维护成本明显更大。

### 决策 2：缩放值采用离散档位并限制范围

- 采用离散档位：`0.8, 0.9, 1.0, 1.1, 1.25, 1.5`。
- 启动恢复和设置写入均执行范围校验，非法值回退 `1.0`。

选择理由：
- 离散档位便于测试矩阵收敛，降低无限连续值导致的边界问题。
- 对用户来说可理解、可预期，且能覆盖主要诉求。

### 决策 3：配置持久化采用现有设置存储链路，不引入新依赖

- 前端设置模块新增 `uiScale` 字段。
- 应用启动时读取配置并执行一次缩放同步。

选择理由：
- 复用既有配置读写逻辑，减少引入新故障面。
- 符合“无新增外部依赖”和“最小改动”原则。

### 决策 4：关键布局采用“可读性优先”回归基线

- 缩放后允许容器重排或滚动，但禁止核心交互控件不可见/不可点击。
- 对固定宽高组件增加必要的最小宽度或溢出策略修正。

选择理由：
- 用户价值优先是可读与可操作，而不是像素级静态对齐。

## Risks / Trade-offs

- [固定像素组件在高缩放下截断] → 建立关键页面缩放回归清单，逐页修正最小宽度与 overflow 策略。  
- [启动阶段先渲染后缩放导致闪动] → 启动时优先读取并尽早应用缩放，减少首帧错位窗口。  
- [跨平台缩放行为差异] → 将缩放能力封装为统一接口，并在 macOS/Windows 做最小回归验证。  
- [用户误设极端缩放造成可用性下降] → 提供“恢复默认”快捷入口并在设置页保留可见提示。  

## Migration Plan

1. 增加配置模型字段 `uiScale`（默认 `1.0`）并提供读取/写入通路。
2. 增加 Tauri 缩放桥接命令与前端调用封装。
3. 在设置页提供档位选择与“恢复默认”入口，支持即时应用。
4. 对关键布局执行缩放回归并修正样式边界。
5. 通过 feature flag 控制发布，灰度后默认开启。

回滚策略：
- 若线上出现可用性回归，关闭 feature flag，强制回退 `1.0` 并隐藏设置入口。

## Open Questions

- 是否需要在状态栏显示当前缩放百分比（如 110%）以提升可见性？
- 是否在首版即支持快捷键缩放（`Cmd/Ctrl +` / `Cmd/Ctrl -` / `Cmd/Ctrl 0`）？
- 是否需要把“缩放影响范围”明确限定到主工作区，不影响原生系统窗口边框元素？
