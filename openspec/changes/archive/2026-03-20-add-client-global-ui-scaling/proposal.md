## Why

部分用户在高分辨率显示器（如 4K/5K）下使用 Codemoss 时，客户端字体和控件会显得过小，主幕布可读宽度也会显著收窄，导致阅读、编辑和导航效率下降。需要提供应用内可控的全局 UI 缩放能力，让体验不再被硬件分辨率被动决定。

## 目标与边界

- 目标：提供一个用户可配置、可持久化的“全局 UI 缩放”设置，统一影响客户端主要界面（含侧栏、会话区域、幕布区域、设置面板）的视觉尺寸与可读性。
- 边界：本提案只定义“缩放能力与交互契约”，不在本阶段重构所有页面的视觉规范，不改变业务功能逻辑与数据模型。

## 非目标

- 不实现“每个模块单独缩放”或“每个页面独立缩放”。
- 不替代操作系统级 DPI/Display Scaling 设置，仅提供应用内补充控制。
- 不在本提案中引入新的主题系统或字体系统。

## What Changes

- 在客户端设置中新增“全局 UI 缩放”滑动条控件（`80% ~ 260%`，步进 `1%`），并显示当前百分比值。
- 缩放设置采用“调整草稿 -> 点击保存”的交互，保存后在当前窗口立即生效（无需重启）。
- 提供“恢复为 100%”按钮，用于快速回到默认缩放草稿，并可通过保存立即生效。
- 缩放设置支持持久化，重启应用后仍保持用户选择。
- 前后端统一加入范围校验：仅接受 `0.8 ~ 2.6`，非法值在启动恢复时回退到 `100%`。

## 技术方案对比

### 方案 A：基于 WebView/Window 的全局 zoom factor（推荐）

- 优点：语义直接，天然覆盖大多数 UI 元素；实现路径清晰；与“全局缩放”目标一致。
- 缺点：对少量基于固定像素/transform 的局部样式可能产生放大后边界问题，需要补齐回归验证。

### 方案 B：仅通过 CSS 根字号与局部容器比例联动

- 优点：前端可控粒度高，可按区域精细调整。
- 缺点：维护成本高，容易出现组件缩放不一致；对图标、间距、弹层定位等需要额外适配，长期复杂度更高。

### 方案 C：依赖操作系统显示缩放，不做应用内设置

- 优点：零开发成本。
- 缺点：无法满足“应用独立可控”的诉求；同机多应用体验会被系统全局策略绑定，不可接受。

取舍：选择方案 A 作为主路径，必要时辅以少量局部 CSS 修正，兼顾一致性与可维护性。

## Capabilities

### New Capabilities

- `client-global-ui-scaling`: 定义客户端全局 UI 缩放的设置入口、生效规则、持久化与回退行为。

### Modified Capabilities

- 无

## 验收标准

- 用户可在设置中看到并调整“全局 UI 缩放”滑动条（`80% ~ 260%`）并看到实时百分比。
- 用户点击“保存”后，1 秒内完成当前窗口缩放更新，无需重启应用。
- 重启应用后缩放设置保持不变。
- 在 80%~260% 支持范围内，关键模块文本可读、主要按钮可点击、无核心信息被遮挡。
- 用户可通过“恢复为 100%”快速回到默认值并保存生效。

## 风险与回滚

- 风险：部分固定宽度/高度组件在高缩放下出现截断或溢出。
- 控制：对关键页面建立缩放回归清单（侧栏、会话列表、幕布、设置页、弹窗）。
- 回滚：保留 feature flag，可在问题版本中临时关闭“自定义缩放入口”，回退到 100% 默认缩放。

## Impact

- Affected specs: 新增 `openspec/specs/client-global-ui-scaling/spec.md`。
- Affected frontend areas（预期）:
  - `src/features/settings/*`（新增缩放设置入口与状态持久化）
  - `src/components/layout/*`（关键布局在不同缩放档位的兼容性）
  - `src/styles/*`（少量缩放兼容修正）
- Affected backend/runtime areas（预期）:
  - Tauri window/webview 缩放桥接与配置读写链路
- Dependencies:
  - 无新增外部依赖（优先复用现有配置存储与窗口控制能力）
