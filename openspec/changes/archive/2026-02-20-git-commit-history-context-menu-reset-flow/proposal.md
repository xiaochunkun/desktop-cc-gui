## Why

当前 `git commit` 历史列表只有基础右键动作，缺少你重点要求的两条高频链路：`复制修订号` 与 `将当前分支重置到此处...`
。这会导致用户需要跳转其他区域或命令行完成关键操作，路径割裂且易误操作。

## 目标与边界

- 目标
    - 在提交列表项提供稳定的右键菜单入口，支持动作分组与可预测排序。
    - 明确落地两条主链路：`复制修订号`、`将当前分支重置到此处...`。
    - 将图3的重置确认弹窗纳入规范：展示目标提交信息与四种模式 `soft/mixed/hard/keep`。
    - 让右键菜单动作与提交区按钮组共享同一套可用性规则（关联按钮组）。
- 边界
    - 本次只做 OpenSpec 提案，不直接改业务代码。
    - 不一次性覆盖图2中所有高级历史改写操作（如 fixup/squash/interactive rebase）。
    - 不改动 Git 仓库模型与底层数据结构。

## 非目标

- 不新增独立 Git 子页面。
- 不引入新的 Git 引擎依赖。
- 不改变现有提交历史列表的分页与渲染策略。

## What Changes

- 扩展提交列表右键菜单为“分组动作菜单”，并要求排序稳定。
- 新增右键动作：`复制修订号`、`将当前分支重置到此处...`。
- 新增“提交区关联按钮组”规范，按钮组与右键菜单动作共用可用性与禁用逻辑。
- 新增“重置确认弹窗”规范：
    - 展示当前分支、目标提交短哈希、subject、author。
    - 支持 `soft/mixed/hard/keep` 模式选择并提供模式解释。
    - 高风险路径（如 `hard`）要求破坏性确认与失败可读反馈。
- 补充重置操作并发锁定约束，避免与其他写操作冲突。

## 技术方案对比

### 方案 A：仅新增右键菜单，不做按钮组联动

- 优点：实现快、改动小。
- 缺点：菜单与按钮状态容易漂移，用户会遇到“菜单可点但按钮禁用”的不一致。
- 结论：不采纳。

### 方案 B：动作注册表统一驱动（采纳）

- 优点：右键菜单和按钮组共享 action id、可见性、禁用原因和执行器，行为一致、易测试。
- 缺点：需要一次性整理现有动作定义。
- 结论：采纳。

### 方案 C：把所有逻辑下沉后端返回菜单结构

- 优点：前端更轻。
- 缺点：交互迭代成本高，前端状态与焦点控制变复杂。
- 结论：不采纳。

## Capabilities

### New Capabilities

- 无

### Modified Capabilities

- `git-commit-history`: 提交列表右键菜单升级为分组动作，并补齐复制修订号与重置入口。
- `git-history-panel`: 提交区按钮组与右键菜单动作关联，约束选中态与禁用态一致。
- `git-operations`: 新增“将当前分支重置到指定提交”操作语义与确认流程。

## 验收标准

- 右键提交列表项时，菜单必须出现并包含 `复制修订号` 与 `将当前分支重置到此处...`。
- 菜单动作分组与排序稳定，不因提交项不同而发生无规则抖动。
- 触发重置动作后，必须先出现确认弹窗，不允许直接执行 reset。
- 弹窗必须提供 `soft/mixed/hard/keep` 四种模式与解释文案。
- 选择 `hard` 时必须出现破坏性提示；取消后不得产生副作用。
- 提交区按钮组与右键菜单在相同上下文下禁用/启用状态一致。
- 任一写操作进行中，reset 动作必须禁用并给出可读原因。
- 失败场景（冲突/仓库状态不允许）必须给出可读错误并保留重试入口。
