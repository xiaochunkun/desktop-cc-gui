## Context

Wave1 已完成 `>3000` 行治理并开启 CI hard gate。  
当前存在一批 `2500-3000` 近阈值文件，但本阶段不再执行 Wave2 批量预防拆分，改为 Deferred + JIT 策略。

## Goals / Non-Goals

**Goals**
- 保持 `>3000` hard gate 的强约束，防止失控增长。
- 建立触线即拆（JIT）规则，确保超阈值变更不带病合并。
- 以最小治理成本维持可维护性与交付节奏。

**Non-Goals**
- 不执行 Wave2 全量近阈值拆分。
- 不做功能扩展与视觉改版。
- 不做一次性全仓重写。
- 不修改持久化格式或跨端协议。

## Decisions

### Decision 1: 单强制阈值 + JIT 治理
- `>3000`: hard gate（强制）
- `2500-3000`: 观察区（信息性，不强制拆分计划）

### Decision 2: 触线同 PR 修复
- 任一 PR 若引入/增长为 `>3000`，必须在同一 PR 内通过拆分或模块抽取得到降线。
- 不允许“先合并超阈值、后续再补拆分”。

### Decision 3: 维持 facade 稳定导出
- JIT 拆分仅改变内部结构。
- 入口文件保留兼容导出与调用约定。

## Risks / Trade-offs

- 风险：触线时出现临时拆分压力  
  缓解：高频文件在日常改动中提前预留模块缝（adapter/facade + 子模块目录）。

- 风险：触线修复 PR 变大导致回归面扩大  
  缓解：限制 JIT 拆分范围为“仅降线所需最小集合”，并补齐目标模块回归验证。

- 取舍：放弃预防性批量治理会损失一部分前置收益  
  缓解：保留观察清单，按热点趋势进行轻量提醒。

## Migration Plan

1. 保持现有 `>3000` hard gate 与扫描脚本不变。
2. 在治理文档中明确“触线同 PR 拆分”与验证门禁。
3. near-threshold 清单作为观察信息维护（可选），不作为强制实施队列。

## Deferred Strategy Review (2026-03-16)

### Deferred 原因
- 当前 `>3000` 已清零，继续推进 Wave2 批量拆分的边际收益下降。
- 批量拆分会增加并行改动冲突半径，且与现阶段功能交付节奏存在竞争。

### JIT 边界
- 唯一强制门禁为 `>3000` hard gate。
- 任一触线 PR 必须同 PR 内完成降线，不允许“先合并后补拆分”。
- 近阈值 `2500-3000` 仅做观察，不构成独立阻断条件。

### 残余风险
- 高频近阈值文件在需求堆叠时仍可能突然触线，导致当期 PR 体量变大。
- 缓解方式：控制 JIT 拆分范围为最小集合，并保持 facade 兼容与目标模块测试。
