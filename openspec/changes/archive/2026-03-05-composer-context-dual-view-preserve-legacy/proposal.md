## Why

当前 Composer 上下文信息仅有单一视图，用户在“保持原有使用习惯”与“获得更清晰的新信息表达”之间只能二选一。  
在不替换旧视图的前提下新增并排视图，可以实现渐进升级：一方面维持现网稳定与认知连续性，另一方面为后续信息架构优化提供低风险落点。

## 目标与边界

- 目标：在 Composer 中引入“并排新视图”，与旧视图同时存在。
- 目标：旧视图默认行为不变（渲染、交互、数据来源、文案语义保持兼容）。
- 目标：支持通过开关控制新视图启用，便于灰度和回退。
- 边界：本次仅新增展示层能力与最小状态适配，不调整后端 token 事件协议。
- 边界：不改动会话压缩、会话恢复、归档等核心线程状态机语义。

## 非目标

- 不重写旧视图组件，不做“新视图替换旧视图”。
- 不引入新的 token 计量口径、计费逻辑或后端字段。
- 不在本次将新旧视图抽象为统一组件框架。
- 不修改 Composer 之外的页面信息架构（Git 面板、Spec Hub、Settings 等）。

## What Changes

- 新增 Composer 并排视图容器编排能力（旧视图 + 新视图并列渲染）。
- 新增新视图组件（只读展示层），消费现有 `latestTokenUsageInfo` 与线程状态。
- 增加新视图可见性控制策略（feature flag / 配置开关）。
- 新增状态展示覆盖：
  - token usage 有值/无值
  - context compacting / compacted
  - loading / empty
- 增加兼容性保护：新视图关闭时，旧视图路径与当前版本行为一致。
- 增加回归测试与视觉回归检查点（重点覆盖旧视图不回退）。

## 技术方案对比与取舍

| 方案 | 描述 | 优点 | 风险/成本 | 结论 |
|---|---|---|---|---|
| A. 在旧组件内直接扩展并排布局 | 复用旧组件并在内部拼接新结构 | 开发快 | 耦合高，旧逻辑被动改动多，回归面大 | 不选 |
| B. 新旧双组件并列 + Adapter 共享状态 | 父容器负责布局，旧组件不改或最小改；新组件独立实现 | 回归风险最低、可灰度、可回滚 | 初期存在少量展示逻辑重复 | 选用 |
| C. 全量重构为统一 Context View 框架 | 从架构层统一新旧视图 | 长期最优 | 当前成本高、交付慢、风险高 | 本期不选 |

取舍：采用 **B**。核心策略是“新增而非替换、隔离而非侵入”，优先保证旧视图稳定。

## 迁移与回滚策略

- 迁移：默认保持旧视图路径不变，新视图通过开关逐步放量。
- 回滚：关闭新视图开关即可回退至旧视图单路径，不依赖后端变更回滚。
- 观测：上线后关注渲染稳定性、交互错误率、用户停留与点击行为（若埋点已存在则复用）。

## 风险与缓解

| 风险 | 说明 | 缓解措施 |
|---|---|---|
| 旧视图行为回退 | 并排编排影响旧组件布局/交互 | 容器隔离、旧组件 API 不变、回归测试强约束 |
| 双视图状态不一致 | 新视图与旧视图显示时序差异 | 统一通过 Adapter 读取同源状态快照 |
| 小屏布局拥挤 | 并排导致可用空间下降 | 增加响应式断点策略（小屏降级为单列） |

## Capabilities

### New Capabilities
- `composer-context-dual-view`: Composer 支持旧视图与新视图并排共存，并可通过开关控制启用。

### Modified Capabilities
- （无）

## 验收标准

- 功能兼容
  - 启用新能力时：旧视图与新视图可并排渲染。
  - 禁用新能力时：行为与当前版本一致（功能与交互无差异）。
- 状态正确性
  - 新视图在 `latestTokenUsageInfo` 有值/无值、`compacting/compacted` 场景下展示正确。
- 协议稳定性
  - 不新增/修改后端 token usage 协议字段；仅消费既有数据。
- 回归保障
  - 通过旧视图主路径回归测试。
  - 通过新视图启用路径与开关切换路径测试。
- 可运维性
  - 提供开关级回退能力（无需后端联动）。

## 改动体量评估

- 总体规模：**中小（Small-Medium）**。
- 主要工作量：
  - 前端容器编排与响应式处理：中
  - 新视图展示组件与状态映射：中
  - 测试补齐与回归验证：中
- 非工作量热点：后端协议、存储结构、线程核心状态机（本次不动）。

## Impact

- 受影响模块（预期）：Composer footer/status 容器、上下文视图组件、相关测试文件。
- API/协议影响：无新增 API；继续使用现有 `thread/tokenUsage/updated` 驱动数据。
- 依赖影响：无新增外部依赖（预期）。
- 工程影响：增加一个可控的新展示面，为后续逐步替换提供演进路径。

## 追加提案（2026-03-05）：边界收敛为 Codex-only

### 追加背景

当前实现已完成双视图能力，但本次交付需要进一步收敛影响面，确保该能力只在 `codex` 引擎界面生效，不改变其它引擎路径和全局设置体验。

### 追加目标

- 仅在 `codex` 引擎下显示新上下文视图（旧指示器在 codex 场景隐藏）。
- 非 `codex` 引擎严格保持现状（UI、行为、配置入口均不变化）。
- Codex 场景总览采用“圆圈占比 + 悬浮气泡明细”形式，不展示额外进度条。
- 继续使用已有事件与状态（`thread/tokenUsage/updated`、`token_count`、`thread/compacted`），不改后端协议。

### 追加非目标

- 不新增全局设置开关（避免对其它引擎形成可见影响）。
- 不修改线程核心状态机，不更改压缩触发机制。
- 不调整 `.codex` 会话落盘结构与上下文传输协议。

### 追加验收标准

- 当 `selectedEngine !== codex` 时，新视图完全不渲染。
- 当 `selectedEngine === codex` 时，渲染新视图并隐藏 codex 场景的 legacy 圆圈。
- 悬浮气泡包含清晰明细（总消耗 + 上下文占用 + 压缩状态文案），且无额外进度条。
- 不引入 settings/tauri 的新配置字段，不改变非 codex 设置面板内容。
