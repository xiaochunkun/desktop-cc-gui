# Design: OpenCode Chat Send Guard and Panel Onboarding

## Context

OpenCode 已具备会话、模型、provider 能力，但当前 UI 把“配置操作”和“消息发送”混在同一视觉层，导致两类问题：

- 错误路径未被阻断（红色连接状态仍可发送）。
- 正确路径不够聚焦（chat 下方出现模型配置控件）。

本次设计目标是把发送路径变成“状态安全默认值”，把配置路径集中到面板，并在面板首页先建立连接认知。

## Goals / Non-Goals

**Goals**

- 红色连接状态下，OpenCode chat 发送必须阻断。
- 管理面板弹层锚点与触发按钮保持空间连续性（按钮上方）。
- OpenCode 模型选择从 chat 区域下沉到管理面板。
- 面板首页提供连接引导与可观测状态摘要（状态/认证/模型数量）。

**Non-Goals**

- 不修改 OpenCode 后端协议与 CLI 调用参数。
- 不引入新的 provider 或认证机制。
- 不改变 Claude/Codex/Gemini 的现有 chat UI。

## Decisions

### Decision 1: 发送护栏在 composer 入口统一裁决

在 OpenCode 发送入口统一判断连接状态，当状态为 red/disconnected 时直接拒绝发送并展示可操作提示。

**Reason**

- 防止不同发送路径（点击发送、Enter、queue）出现规则不一致。
- 护栏应在最靠近用户动作的位置执行，减少无效请求。

### Decision 2: 管理面板弹层改为“按钮上方锚定”

所有从 OpenCode 管理按钮触发的主弹层，默认在触发按钮上方打开；空间不足时再做受控回退。

**Reason**

- 强化触发源与弹层的空间关联。
- 减少“右上角跳出”的视觉断裂与寻址成本。

### Decision 3: 模型选择归位到管理面板

chat 输入区仅保留发送必需控制；OpenCode 模型选择统一放入管理面板，避免在聊天主链路出现配置噪音。

**Reason**

- 聊天与配置职责分离。
- 降低 composer 高度与视觉负担。

### Decision 4: 管理面板首页无默认连接 + onboarding 摘要

首页不自动选中默认连接，而是显式展示：

- 当前连接状态
- 已完成认证类型
- 可用模型数量
- 下一步建议操作

**Reason**

- 让“当前状态”优先于“默认值假象”。
- 新用户可直接知道系统是否可发送、差什么认证步骤。

## Risks / Trade-offs

- 护栏过严可能阻断临时抖动场景。缓解：提供“重试检测连接”快速动作。
- 模型选择迁移后，老用户会有短期适应成本。缓解：首次迁移提示与入口高亮。
- 弹层上方锚定在小窗口中可能遮挡。缓解：保留空间不足回退策略。

## Validation

- 手动验证：红色状态下，发送按钮和 Enter 均无法发送。
- 组件测试：OpenCode 模型选择区不在 composer 下方渲染。
- 视觉验证：管理面板从按钮上方弹出，位置稳定。
- 状态验证：首页在无默认连接时仍完整展示状态/认证/模型数量。
