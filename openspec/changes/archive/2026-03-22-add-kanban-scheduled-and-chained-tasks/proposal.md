## Why

初版提案已覆盖“定时 + 串联”的核心方向，但在连续迭代后，真实代码行为已经明显扩展（调度治理、链路触发门禁、分组展示、倒计时与暂停、卡片标签体系等）。若不及时同步提案，会导致“文档范围”与“已上线行为”脱节，后续评审和归档都容易误判。

本次更新目标：将提案对齐到当前已实现代码基线，明确哪些能力已落地、哪些仍是边界。

## What Changes

- 调度模型已落地 `manual / once / recurring`，并支持 recurring 的两种执行方式：
  - `same_thread`（同会话）
  - `new_thread`（每次新会话）
- recurring 执行语义收敛为“完成后推进下一周期”，避免同一窗口重复触发；应用离线期间不做补跑，仅计算未来窗口。
- 为待办中的调度任务增加治理能力：暂停/恢复调度、取消调度、秒级倒计时、暂停后倒计时冻结。
- 调度状态标签按列语义展示（调度器 / 执行中 / 已调度），并对阻断原因做标准化映射。
- 串联任务收敛为线性链：仅允许 todo 节点建链，禁止多分支与循环，禁止下游保留独立计划。
- 串联触发门禁增强：非链头任务禁止手动/拖拽单独执行，只允许由上游成功后自动触发。
- 新增拖拽约束：非关联任务不允许拖入关联组内部槽位。
- recurring + new_thread 场景下，完成一次执行后会产出新的待办实例，历史实例进入审查/完成列，保持“调度源任务”和“执行实例”分离。
- 审查区/完成区支持分组折叠（周期新会话组、关联任务组），并支持组内自动排序。
- 关联组补充可识别元数据：
  - 3 位组码（同组一致）
  - 组内串行顺序标号
- 卡片元信息标签改为分类着色（计划类 / 时间类 / 频率倒计时类 / 执行模式类 / 链路与序号类）。
- 执行时间追踪语义明确：进入执行态写入开始时间，离开执行态写入结束时间，审查/完成卡片可见。
- 清理无效噪音标签（例如“结果快照”展示标签），保留必要的阻断与调度信息。
- **无 BREAKING 变更**：普通任务创建、拖拽启动、会话联动路径保持兼容。

## 目标与边界

- 目标
  - 让 Kanban 从“手动触发入口”升级为“可治理的轻量编排入口”。
  - 让周期任务和线性串联在同一状态机内稳定运行，避免多触发源冲突。
  - 让用户在看板层清晰识别：当前是调度源任务、执行实例、还是链路节点。
- 边界
  - 仅保证应用运行期间调度；不承诺进程退出期间准时触发。
  - 仅支持线性链，不引入 DAG/并行分叉/条件路由。
  - 结果传递仍以结构化摘要为主，不搬运完整历史会话。

## 非目标

- 不实现系统级 cron daemon 或云端托管调度。
- 不实现任务失败重试策略中心、补偿事务、优先级抢占。
- 不做跨 workspace 的串联编排。
- 不重写现有线程存储或执行引擎。

## 兼容性边界保证

- `schedule / chain / execution / lastResultSnapshot` 保持 optional + additive，旧任务可直接读取。
- 状态语义维持 `todo / inprogress / testing / done`，新增能力不重定义状态机。
- 非调度普通任务不展示调度治理动作，不引入额外阻断入口。
- 链路非法配置、规则无效或触发冲突时，系统必须阻止执行并输出明确原因，不得破坏既有数据。
- 重启后分组展示需保持稳定（基于 groupId/seriesId 与回退策略聚合），避免历史执行实例被错误分裂。

## 连续执行场景（更新）

- **周期 + 新会话**：调度源任务在 `todo` 保持可见并等待下一次触发；每次执行生成新的实例卡片，实例完成后进入 `testing/done`，并带轮次编号。
- **周期 + 同会话**：同一任务在轮次内循环执行，达到轮次上限后自动停止并标记原因。
- **线性串联**：`A -> B -> C` 中，只有 `A` 可作为人工/计划入口；`B/C` 只能由上游成功后自动触发。
- **阻断治理**：若上游无可用快照、下游状态非法或被手动取消调度，下游保持待办并记录阻断原因，等待人工介入。
- **拖拽约束**：用户不能将非关联任务拖入现有关联组内部位置，避免组结构被破坏。

## 技术方案对比

### 方案 A：应用内调度器 + 线性任务链（已采用）

- 做法：在现有 Kanban store 上扩展调度/链路元数据；统一执行入口承接 `manual/drag/autoStart/scheduled/chained`；用运行时扫描推进任务。
- 优点：复用现有执行链路，改动集中，可快速迭代并保持可回滚。
- 缺点：应用关闭期间无法触发；高级编排能力有限。

### 方案 B：独立后端编排服务

- 做法：引入常驻调度与通用依赖图引擎。
- 优点：能力上限更高。
- 缺点：复杂度与回归成本显著上升，超出当前产品阶段。

### 取舍

继续采用方案 A，确保“功能密度提升”与“系统可控复杂度”之间平衡。

## Capabilities

### New Capabilities

- `kanban-task-scheduling`: 支持一次性/周期性调度、倒计时、暂停恢复、触发门禁与运行态反馈。
- `kanban-task-chaining`: 支持 todo 线性串联、上游结果传递、自动续跑与阻断治理。

### Modified Capabilities

- 本次变更为上述能力的范围扩展与交互细化，不新增第三个 capability。

## 验收标准（同步到当前实现）

- 用户可配置 `manual / once / recurring`；recurring 可选 `same_thread / new_thread` 与结果传递策略。
- 到期调度任务可后台自动触发，运行中任务不会重复触发；离线错过窗口不补跑。
- 待办中的调度任务支持暂停/恢复与取消调度；倒计时为秒级且暂停后冻结。
- 链路仅允许 todo 节点，禁止循环与多分支；下游禁止保留独立调度计划。
- 非链头任务不能手动或拖拽单独执行，必须由上游自动续跑触发。
- 非关联任务不能拖入关联组内部；组内任务按链路顺序自动排序。
- 关联组具备统一 3 位组码，组内任务具备顺序标号，卡片可见。
- 审查区/完成区支持分组折叠；执行实例显示开始时间与结束时间。
- 标签按类别着色，阻断原因可读且一致；无意义标签不再展示。
- 旧任务（无新增字段）可直接读写并保持原有普通任务行为不回归。
