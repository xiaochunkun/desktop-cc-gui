## Context

当前文件查看器已经具备通用文件打开、多 Tab、文本预览、CodeMirror 编辑和 Markdown 渲染能力，但这几条链路的体验契约没有统一：

- 文本预览与编辑区域主要依赖现有表面色和 `theme="dark"` 的编辑器主题，导致浅色/深色场景下正文对比度都不够稳定。
- Markdown 文件在编辑态默认进入双栏“源码 + 实时预览”，更像工作台而不是阅读器，首次打开时认知负担偏高。
- 长行策略没有被当作一等设计约束处理。源码视图、Markdown 结构块与部分渲染容器之间，对“自动换行”和“横向滚动”没有统一边界。

这次变更是典型的跨组件前端交叉改动：`FileViewPanel` 管模式切换，`file-view-panel.css` 管文件查看器主题和布局，`Markdown.tsx` 与 `messages.css` 管 Markdown 渲染子结构。虽然不涉及后端，但如果不先写清设计，后续实现容易只修样式、不修交互契约。

## Goals / Non-Goals

**Goals:**

- 让文本类文件在浅色与深色主题下都具备稳定且明确的阅读对比度。
- 将 Markdown 默认进入模式改为单屏源码编辑，降低首次打开认知负担。
- 将“长行不强制折行，优先横向滚动”变成文件查看器的显式约束。
- 在不新增依赖、不更改文件树打开契约的前提下完成体验收束。

**Non-Goals:**

- 不新增 binary 文件 viewer。
- 不改变后端文件读取、截断、binary 判定、编码探测逻辑。
- 不顺带统一聊天消息区和其他业务区域的 Markdown 风格，只复用已有通用样式能力。

## Decisions

### Decision 1: 文件查看器采用主题感知的文本对比度基线，而不是继续绑定固定深色编辑器视觉

现状里文件查看器的 CodeMirror 编辑器通过 `theme="dark"` 固定在深色视觉上，而周围容器又依赖全局主题变量，造成浅色主题下局部视觉不统一，深色主题下正文层级又偏灰。

采用方案：

- 以文件查看器容器为主，统一定义浅色/深色两套正文区域基线。
- 让预览态与编辑态共享同一组“正文背景 / 主文本 / 行号 / 高亮辅助色”语义，不再让预览和编辑各自漂移。
- CodeMirror 主题以“跟随文件查看器主题”为目标，而不是继续把编辑器视为独立黑盒。

备选方案：

- 保留固定 dark editor，仅微调 preview 颜色。

不采用原因：

- 只能修掉一半问题，浅色主题仍会出现编辑态割裂。
- 后续每次调 token 颜色都需要同时兼容两个互相独立的视觉系统，维护成本更高。

### Decision 2: Markdown 将现有“编辑态内嵌实时预览”改为与通用文件模式对齐的单屏切换

现有 Markdown 编辑态使用 split-pane：左侧 CodeMirror，右侧 `Markdown` 组件实时渲染。这个设计在写作时有价值，但它不适合作为默认打开体验，因为用户从文件树进入时的主诉求往往是阅读源码或快速修改，而不是一上来就进入双栏工作台。

采用方案：

- Markdown 默认进入 `edit`，且 `edit` 只展示源码编辑器。
- Markdown 的 `preview` 作为显式模式展示渲染结果。
- 顶部动作区对 Markdown 仅保留“编辑 / 预览”这组主切换，不再把源码/渲染态拆成第二层模式。

备选方案：

- 保留双栏，但记住用户上次选择。
- 默认预览，编辑时再切回双栏。

不采用原因：

- “记住上次选择”会把复杂度继续留在模式系统里，不能解决默认态过重的问题。
- 默认预览不符合你当前需求，也会拉高修改路径成本。

### Decision 3: 将 no-wrap 作为源码视图默认策略，将横向滚动作为宽内容的主退路

源码视图中的自动换行会破坏缩进、长命令、长 JSON、表格文本以及 Markdown 原始结构的判断。真正该折中的不是内容本身，而是容器的滚动能力。

采用方案：

- 文件查看器的源码类视图默认关闭自动换行。
- 预览态的逐行代码渲染与编辑态的 CodeMirror 都必须支持横向滚动。
- Markdown 渲染态不要求把普通 prose 段落强行改成单行，但宽结构内容必须禁止被强制折行破坏，并为其提供横向滚动容器。

备选方案：

- 全量开启自动换行，只为 code block 单独开滚动。
- 允许用户在查看器里切换 wrap/no-wrap。

不采用原因：

- 第一种不能解决源码与长结构内容的核心痛点。
- 第二种会把本次本该固定下来的阅读契约再次变成“用户自己救自己”的设置问题，超出当前需求边界。

### Decision 4: 优先复用现有 Markdown 组件与样式钩子，不新增独立 Markdown viewer

文件查看器已经复用消息区的 `Markdown` 组件。问题主要在容器约束和样式语义，而不在渲染栈本身。

采用方案：

- 继续复用现有 `Markdown` 组件。
- 在文件查看器命名空间下补充更明确的 Markdown 容器规则，必要时只对宽结构块补充 file-view 级样式。

备选方案：

- 为文件查看器单独实现一套 Markdown 组件。

不采用原因：

- 容易造成两套 Markdown 行为再次分叉。
- 对当前目标来说成本过高，且没有明确收益。

## Risks / Trade-offs

- [Risk] Markdown 取消默认双栏后，少数依赖“边写边看”的用户会感知功能后退
  → Mitigation：保留显式预览切换，且实现阶段可评估是否保留快速键盘切换能力。

- [Risk] 调整 CodeMirror 与 preview 颜色后，现有 token 颜色在浅色主题下可能出现部分可读性不足
  → Mitigation：把主文本对比度与 token 对比度分层处理，先保证正文与背景，再单独检查常用 token 组。

- [Risk] “不要自动换行”如果被扩大解释到所有渲染段落，可能导致 Markdown prose 在窄视口下横向滚动过多
  → Mitigation：设计上将 no-wrap 明确限定为源码视图与宽结构内容，普通渲染段落保持标准 Markdown 阅读习惯。

- [Risk] 文件查看器与消息区共用 Markdown 样式片段时，局部样式覆盖可能互相影响
  → Mitigation：优先在 `fvp-` 命名空间下收口覆盖，避免修改全局 `.message` 规则的默认行为。

## Migration Plan

1. 先调整文件查看器的模式状态机，使 Markdown 默认进入单屏编辑，并把顶部切换简化为编辑 / 预览。
2. 再统一文件查看器的文本主题变量与 CodeMirror 样式，使浅色/深色主题下的正文对比度达标。
3. 最后处理 no-wrap 与横向滚动细节，重点验证源码视图、Markdown 宽结构块、代码块、表格和长链接。
4. 回滚策略保持简单：若新交互出现回归，可独立回退文件查看器模式切换层或样式层，而不影响文件树打开和后端读文件逻辑。

## Open Questions

- 当前不计划为 Markdown 保留“渲染态中的 source 子模式”。如果实现中发现有明确使用场景，再作为后续独立 change 处理。
- 如果浅色主题下现有 Prism / CodeMirror token 颜色与新背景冲突，是否需要在本次内顺带定义 file-view 专属 token palette，留待实现阶段评估。
