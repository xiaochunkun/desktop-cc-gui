## Why

当前文件查看器的文本类文件阅读体验存在三个直接问题：一是文本与代码内容在现有主题下对比度不足，浅色与深色场景都偏灰，长时间阅读吃力；二是 Markdown 文件默认进入双栏编辑模式，首次打开时信息密度过高，不符合“先看源码、按需预览”的常见心智；三是 Markdown 渲染与源码查看中的自动换行会破坏段落、表格、代码块和长行结构，影响对原始内容的判断。

这个问题需要现在处理，因为右侧文件树已经成为通用文件入口，文件“能打开”已经不是主要矛盾，真正影响可用性的瓶颈已经转移到“打开后是否好读、好看、好切换”。

## 目标与边界

- 目标是提升文件查看器内文本类文件与 Markdown 文件的阅读清晰度、模式切换直觉性与长行可读性。
- 变更边界仅覆盖文件查看器相关交互与渲染契约，包括文本预览、文本编辑、Markdown 编辑、Markdown 预览以及它们的主题样式和滚动策略。
- 本提案不改变文件树打开契约，不改变后端读文件策略，不引入新的文件类型 viewer。

## 非目标

- 不新增 PDF、Office、音视频等 binary 文件的专属预览器。
- 不重做聊天消息区、评审区、Spec Hub 等其他 Markdown 消费场景。
- 不在本次变更中扩展新的语言识别规则或新增文件格式支持范围，除非为了保持 Markdown/source 渲染契约一致而必须补齐。

## What Changes

- 调整文本类文件在文件查看器中的主题对比度策略：
  - 浅色主题下使用浅底深字。
  - 深色主题下使用深底浅字。
  - 预览与编辑两条链路保持一致的可读性基线，而不是继续使用偏灰的默认文本层级。
- 调整 Markdown 文件默认打开行为：
  - 默认进入单屏源码编辑模式。
  - 不再默认展示双栏“编辑 + 实时预览”布局。
  - 顶部按钮层提供清晰的编辑 / 预览切换。
- 优化 Markdown 文件在文件查看器中的渲染契约：
  - 统一源码查看与渲染态的版式边界。
  - 关闭自动换行，长内容通过横向滚动查看。
  - 重点保证代码块、表格、长链接、长段落在不折行前提下保持结构稳定。
- 为本次交互调整补充非回归约束，确保现有文件树多 Tab 打开能力与既有已支持语言的基础渲染不被破坏。

## Capabilities

### New Capabilities

- `file-view-markdown-reading-ux`: 定义 Markdown 文件在文件查看器中的默认进入模式、编辑/预览切换方式，以及 Markdown 专属阅读布局契约。

### Modified Capabilities

- `file-view-language-rendering-coverage`: 调整文本类文件在预览与编辑态的主题可读性要求，并补充“关闭自动换行、改为横向滚动”的显示契约。

## Impact

- Affected frontend:
  - `src/features/files/components/FileViewPanel.tsx`
  - `src/styles/file-view-panel.css`
  - `src/features/messages/components/Markdown.tsx`
  - `src/styles/messages.css`
- Affected behavior:
  - 文件查看器中的纯文本 / 代码预览
  - 文件查看器中的 CodeMirror 编辑器视觉主题
  - Markdown 文件默认打开模式
  - Markdown 渲染与长行滚动策略
- APIs / backend:
  - 无新增后端接口
  - 不修改现有文件读取、截断、binary 判定逻辑
- Dependencies:
  - 优先复用现有 CodeMirror、React Markdown 与现有主题变量，不新增依赖

## 技术方案

### 方案 A：局部修补现有双栏与样式

- 保留 Markdown 默认双栏编辑，只微调颜色与局部 CSS。
- 在现有预览容器上追加少量 `overflow-x` 规则，尽量不动交互结构。

优点：

- 改动面最小，短期实现快。

缺点：

- 不能解决默认交互心智错误，用户首次打开 Markdown 仍然先看到双栏。
- 文本主题与 Markdown 主题仍容易分别演进，后续继续漂移。
- 自动换行问题会在编辑态、预览态、嵌套 Markdown code block 中继续反复出现。

### 方案 B：重申文件查看器的文本/Markdown 渲染契约

- 将“文本阅读主题”“Markdown 默认模式”“长行滚动优先”提升为明确能力要求。
- Markdown 默认进入单屏源码编辑；预览改为显式切换，而不是并列常驻。
- 对文件查看器内的文本预览、CodeMirror 编辑与 Markdown 渲染统一约束主题对比度和 no-wrap 策略。

优点：

- 能一次性解决当前三个核心痛点。
- 交互更贴近用户心智，默认态更安静。
- 约束清晰，后续新增语言或微调样式时更不容易回退。

缺点：

- 需要同时调整组件逻辑和 CSS 契约，改动面比纯样式修补更大。

### 取舍

采用方案 B。

原因是这次问题不是单点样式缺陷，而是“默认模式 + 主题对比度 + 换行策略”三个契约同时失衡。继续做方案 A 只能止血，不能解决默认体验错误，也无法保证后续不再次回退。

## 验收标准

- 打开任意文本类文件时：
  - 浅色主题下文件内容区域为浅底深字。
  - 深色主题下文件内容区域为深底浅字。
  - 文本主体不得再使用导致低对比阅读困难的灰色主文本。
- 打开 Markdown 文件时：
  - 默认进入单屏源码编辑模式。
  - 首次打开不出现双栏实时预览布局。
  - 顶部操作区必须可在编辑态与预览态之间显式切换。
- 查看 Markdown 源码与渲染结果时：
  - 文件查看器内默认不自动换行。
  - 超出可视宽度的长行、代码块、表格与长链接可通过横向滚动访问完整内容。
  - 不得因关闭换行导致空白渲染、内容裁切不可达或滚动区域失效。
- 非回归：
  - 文件树多 Tab 打开、激活、关闭行为保持不变。
  - 现有已支持文本文件类型的基础打开与保存链路保持可用。
