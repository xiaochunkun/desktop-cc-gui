## Why

当前项目在 Codex 引擎里已有“上下文占用/压缩状态”的展示能力，但压缩本身仍依赖外部引擎被动回传，缺少本地可控的自动压缩执行链路。随着长会话增长，用户会出现上下文占比持续高位、响应变慢、可用窗口被挤占的问题，需要引入“真实自动压缩能力”来主动降压并稳定对话体验。

## 目标与边界

- 目标：在 `codex` 引擎下实现真实自动压缩流程（触发、执行、状态更新、结果落盘）。
- 目标：压缩后维持会话语义连续，不破坏后续问答上下文。
- 目标：沿用现有展示层（圆圈 + 悬浮明细），让 UI 反映真实压缩执行状态。
- 边界：仅作用于 `codex` 引擎线程；`claude/opencode/gemini` 路径零行为变化。
- 边界：不修改 `.codex/archived_sessions` 既有格式定义；如需附加字段，采用向后兼容扩展。

## 非目标

- 不在本期重构所有引擎的上下文管理机制。
- 不引入新的计费口径或 token 统计来源。
- 不做跨会话知识库摘要系统（仅当前线程压缩）。

## What Changes

- 新增 Codex 自动压缩调度器（runtime 侧）：
  - 监听 `token_count` / `thread/tokenUsage/updated`。
  - 当上下文占比达到阈值（默认 92%）时进入压缩准备。
  - 执行压缩并将上下文占比回落到目标区间（默认 <= 70%）。
- 新增压缩执行状态流：`idle -> compacting -> compacted -> idle`。
- 新增 Codex 专属手动压缩入口：
  - 在 Context usage 悬浮窗提供手动触发 icon 按钮。
  - 点击后主动调用 compaction RPC fallback（与自动压缩共用同一兼容策略）。
  - 入口仅在 `codex` 引擎显示，不影响其他引擎 UI。
- 扩展事件链路（Codex 专属）：
  - 复用现有 `thread/compacted`。
  - 增加可选 `thread/compacting`（或等价内部事件）用于 UI 与状态同步。
- 增加失败保护：压缩失败时回退到未压缩上下文快照，并上报可观测错误信息。
- 增加可配置参数（默认隐藏在实验配置）：阈值、目标占比、最小压缩间隔、失败重试上限。

## 技术方案对比与取舍

| 方案 | 描述 | 优点 | 风险/成本 | 结论 |
|---|---|---|---|---|
| A. 仅 UI 侧“伪压缩” | 只重置展示占比，不执行真实上下文重写 | 改动最小 | 不能降低真实上下文负载，属于假象 | 不选 |
| B. Codex Runtime 原位压缩（选用） | 在 Codex 线程执行摘要化压缩并替换可压缩历史段 | 能真实降上下文负载，且可与现有事件流对齐 | 需要新增状态机与失败回滚 | 选用 |
| C. 全引擎统一压缩框架 | 抽象通用压缩平台覆盖所有引擎 | 长期一致性更好 | 本期范围过大，回归风险高 | 本期不选 |

取舍：本期采用 **B**，先把 Codex 真压缩能力做稳，再评估抽象为全引擎框架。

## 验收标准

- 功能正确性
  - 在 `codex` 线程中，当上下文占比 >= 92% 时，系统自动进入 `compacting`。
  - 压缩完成后触发 `thread/compacted`，上下文占比回落到目标区间（<= 70%）。
- 边界正确性
  - 非 `codex` 引擎在同等输入下，不触发压缩调度逻辑且无行为变化。
- 稳定性
  - 压缩失败时线程可继续对话，不丢失原始会话内容。
  - 同一线程在最小压缩间隔内不会重复触发压缩风暴。
- 可观测性
  - 日志可追踪：触发原因、执行耗时、压缩前后 token 占比、失败原因。
- 回归保障
  - 现有 ContextBar 展示路径与 legacy 指标不回退。

## Capabilities

### New Capabilities
- `codex-context-auto-compaction`: 为 Codex 线程提供真实自动压缩能力（触发/执行/回滚/观测）。
- `codex-compaction-runtime-events`: 提供压缩过程事件与状态同步能力，驱动 UI 与 reducer 一致更新。

### Modified Capabilities
- （无）

## Impact

- 受影响模块（预期）
  - Codex 线程事件接入与状态管理（runtime/event hooks/reducer）。
  - Composer 上下文状态展示读取路径（仅 codex 分支）。
  - 配置读取与默认值模块（仅 codex compaction 参数）。
- API/协议影响
  - 兼容保留既有 `token_count` 与 `thread/compacted`。
  - 可能新增 `thread/compacting` 内部事件（向后兼容，不影响非 codex）。
- 风险与回滚
  - 回滚方式：关闭 Codex 自动压缩开关（runtime flag）即可恢复旧行为。
  - 不依赖数据库迁移，不要求后端协议破坏性变更。
