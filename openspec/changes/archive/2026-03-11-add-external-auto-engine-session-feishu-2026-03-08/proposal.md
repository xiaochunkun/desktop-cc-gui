# Proposal: External Auto Engine Session for Feishu

## Why

当前飞书接入已经支持入站消息、特殊 session 创建和手动回复，但用户仍需在客户端内人工点选动作，无法形成真正的“外部消息驱动 AI 自动协作”链路。这导致飞书消息进入 CodeMoss 后仍存在明显人工断点，无法验证自动选引擎、自动续聊与自动回发这条核心创新路径。

因此需要在既有飞书接入能力之上补充自动引擎选择与自动回复闭环，验证 Phase 2 的产品价值，同时保持手动模式兼容。

## What Changes

- 为 special session 增加自动引擎选择握手状态与路由信息。
- 在飞书 WS 入站事件后自动创建 special session，并回发引擎选择提示。
- 解析用户在飞书中的引擎选择输入，将后续消息自动路由到对应引擎并自动回发结果。
- 补充 auto flow 的幂等保护、审计记录与前端自动刷新能力。

## 目标与边界

在现有“飞书入站 + 手动回复”基础上，新增自动对话链路：

- CodeMoss 收到飞书入站消息后自动创建 special session（若不存在）。
- 自动回复并询问用户选择 `Codex` 或 `Claude Code` 引擎。
- 用户选择后，后续飞书消息自动路由到对应引擎并自动回发到飞书。

## 非目标

- 不引入复杂多租户路由策略（本期仅使用当前可用 workspace）。
- 不改造现有手动回复与本地任务入口（保留兼容）。
- 不实现跨平台 Provider（本期仅飞书）。

## 方案对比

### 方案 A：前端轮询驱动自动流程
- 优点：实现简单，不改 WS runtime 结构。
- 缺点：依赖前端页面存活，后台消息处理不稳定。

### 方案 B：后端 WS 入站事件直接驱动自动流程（本提案）
- 优点：链路闭环，页面无感知也可自动处理；与入站事件一致性更高。
- 缺点：后端状态机复杂度上升，需要补充并发保护。

结论：选择方案 B。

## 验收标准

1. Phase2 开关开启后，飞书首次入站可自动收到“引擎选择”提示。
2. 用户回复 `codex/claude`（或 `1/2`）后，后续消息无需人工点击即可得到 AI 回复。
3. 自动流转全程有审计记录与指标（含失败分支）。
4. 保持原有手动回复与连通性测试能力不回退。
