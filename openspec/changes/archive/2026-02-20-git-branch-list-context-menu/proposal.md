## Why

当前分支列表在每一行尾部都直接展示 `checkout` 按钮，信息密度高但视觉噪声也高，导致分支名、ahead/behind
徽标和当前分支状态不够聚焦。随着分支数量增多，界面可读性与操作一致性明显下降，且难以扩展更多分支级操作。

追加问题（2026-02-20）：菜单项“与 `<current>` 比较”当前仅切换分支选中状态，在目标分支已选中的场景会表现为无变化，用户无法得到“比较结果”这一可见反馈。
追加问题（2026-02-20 / 二次修订）：第一版 compare 虽已可见，但采用“文件差异列表 + 单文件详情”逻辑，不符合 IDEA“Compare
Branches”视图的核心语义（双向提交集合对比）。

## 目标与边界

- 目标
    - 将“分支行尾常驻 checkout 按钮”改为“分支行右键菜单”。
    - 在右键菜单最上层增加“分支跟踪信息”只读展示（形如 `<local> -> <upstream>`），并保持 `Checkout` 为第一可执行项。
    - 在不降低效率的前提下提升分支列表可读性，保证大分支列表场景下操作稳定。
    - 支持按分支类型（本地/远程/当前分支）提供差异化可用动作与禁用态。
    - 修复“显示与工作树的差异”在大变更量场景的卡顿：先展示差异文件列表，再按需加载单文件详情。
    - 将“与 `<current>` 比较”升级为 IDEA 风格的双向提交集合对比视图（`A\B` 与 `B\A` 双列表 + 提交详情）。
- 边界
    - 本次只改“分支列表交互层”，不重构 Git backend command 协议。
    - 本次不新增复杂编排能力（如交互式 rebase 向导、批量分支操作）。
    - 不改变现有分支分组、搜索、ahead/behind 计算逻辑。
    - 差异按需加载仅针对“分支菜单 -> 显示与工作树的差异”链路，不改全局 diff viewer 行为。
    - compare 仅针对同仓库分支，不包含跨仓库对比或 PR 评审流程。

## 非目标

- 不新增独立的“分支详情页”或新面板。
- 不在本次引入跨仓库分支策略模板。
- 不修改现有提交历史（commit log）与 diff 展示架构。

## What Changes

- 分支列表 UI 调整
    - 移除每个分支行尾部常驻 `checkout` 按钮。
    - 分支行支持鼠标右键（contextmenu）触发操作菜单。
    - 保留左键行为用于选中/高亮，不与右键行为冲突。

- 右键菜单信息架构（按图 2，最上层展示 tracking，`Checkout` 第一可执行项）
    - 顶栏（分支跟踪信息，只读）
        - `<local-branch> -> <upstream-branch>`（示例：
          `codex/feat-codex-chat-20260216-v0.1.7 -> origin/chore/bump-version-0.1.7-split`）
        - 当本地分支未设置 upstream 时显示：`<local-branch> -> (未设置上游分支)`
    - 第一组（快速切换）
        - `签出 (Checkout)`（第一可执行项）
        - `从 '<branch>' 新建分支...`
        - `签出并变基到 '<current>'`
    - 第二组（查看与比较）
        - `与 '<current>' 比较`
        - `显示与工作树的差异`
    - 第三组（分支关系操作）
        - `将 '<current>' 变基到 '<branch>'`
        - `将 '<branch>' 合并到 '<current>' 中`
    - 第四组（同步）
        - `更新 (Update)`
        - `推送... (Push)`
    - 第五组（管理）
        - `重命名...`
        - `删除`

- 可用性与安全约束
    - 当前分支上禁用 `Checkout` 与 `删除`。
    - 分支跟踪信息行为为只读，不可点击，不参与菜单动作执行。
    - 本地分支显示真实 upstream 引用；无 upstream 时显示明确占位文案，避免误解为数据缺失。
    - 远程分支上仅展示/启用适配动作（例如 checkout as local、compare、update），不暴露无意义动作。
    - 对破坏性动作（删除、可能覆盖历史的变基/合并）保留二次确认。

- 交互细节
    - 菜单支持键盘导航与 `Esc` 关闭。
    - 菜单定位跟随鼠标点击点，避免遮挡当前分支行关键信息。
    - 长分支名场景下菜单文案截断与 tooltip 行为保持一致。
    - “显示与工作树的差异”弹窗先展示文件名列表；用户点击文件时才请求该文件完整 diff，避免一次性拉取全量内容导致卡死。
    - “与 `<current>` 比较”采用 IDEA 语义：
        - 左/上列表：存在于目标分支但不存在于当前分支的提交。
        - 右/下列表：存在于当前分支但不存在于目标分支的提交。
        - 选择任意一侧提交时，在对应详情区展示该提交的文件列表与提交信息摘要。
        - 在提交详情区点击文件时，弹出单文件变化预览（diff modal）。
    - compare 不再以“文件差异列表”作为主入口，禁止与“显示与工作树的差异”共用同一业务语义。

## 技术方案对比

### 方案 A：保留行尾常驻按钮（现状）

- 优点：实现最简单、用户无需学习新入口。
- 缺点：分支行视觉噪声高；可扩展动作受限；按钮在超长列表中点击密度过高。
- 结论：不采纳。

### 方案 B：分支行右键菜单（本提案）

- 优点：列表更干净；动作可扩展；与专业 Git 工具交互一致；可按分支类型动态呈现。
- 缺点：需要新增 context menu 状态管理与可访问性处理。
- 结论：采纳。

### 方案 C：行尾 `...` 下拉菜单

- 优点：可发现性高于纯右键；改造可控。
- 缺点：仍占用行尾空间；密集列表下易误触；与现有视觉目标冲突。
- 结论：不采纳。

## Capabilities

### New Capabilities

- 无

### Modified Capabilities

- `git-branch-management`: 将分支操作入口从“行尾单按钮”升级为“分支级右键菜单”，并规范菜单动作分组、排序与禁用规则。
- `git-operations`: 补充分支菜单触发 `update/push/rebase/merge` 的交互约束与确认流程。

## Impact

- 前端
    - 分支列表行组件（去除行尾 checkout 按钮、接入 right-click 事件）。
    - 分支上下文菜单组件（顶部 tracking 只读条、动作分组、禁用态、键盘可访问性、菜单定位）。
    - i18n 文案新增/调整（中文与英文动作标签、无上游占位文案）。
- 后端
    - 在保持现有 Git 操作 command 的前提下，保留“显示与工作树的差异”轻量 diff 接口（文件列表 + 单文件详情）。
    - 新增/改造分支 compare 所需查询：双向提交集合（`A\B`、`B\A`）与提交详情查询。
    - 复用现有分支元信息中的 upstream 字段，不新增 tracking 专用 command。
- 测试
    - 分支列表交互测试：右键弹出、动作顺序、禁用态。
    - 回归测试：checkout/create/rename/delete/merge/rebase/update/push 的入口迁移后可用。

## 验收标准

- 分支列表中不再出现每行常驻 `checkout` 按钮。
- 在任意分支行右键可弹出菜单，菜单最上层显示分支 tracking 信息。
- `Checkout` 作为第一可执行项保留在菜单动作区第一位。
- 菜单动作分组与顺序符合本提案（与图 2 一致）。
- 分支无 upstream 时，tracking 区域展示明确占位文案（而非空白）。
- 当前分支的 `Checkout` 与 `删除` 动作为禁用态，且有明确原因提示。
- 本地/远程分支动作集合符合类型约束，不出现无效操作。
- 关键动作（checkout / rename / delete / merge / rebase / push / update）迁移后功能可用且通过现有回归。
- 点击“与 `<current>` 比较”后，必须出现双向提交集合视图（`A\B` 与 `B\A`），而非文件差异列表主视图。
- 任一侧提交被选中时，必须展示该提交对应的文件变更与提交摘要信息。
- 在 compare 提交详情区点击文件后，必须弹出对应单文件 diff 预览，且关闭后返回 compare 工作区上下文不丢失。
