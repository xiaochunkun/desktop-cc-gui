## Context

当前 Git 分支列表在行尾常驻 `checkout` 按钮，造成以下问题：

- 视觉上每行都存在高权重操作控件，分支名和状态信息被稀释。
- 交互上只暴露了单动作入口，不利于承载更多分支级操作。
- 随着分支数量增加，列表密度与可读性下降，误触概率上升。

现有代码基线已具备以下前提：

- 分支数据、当前分支状态、ahead/behind 状态可稳定获取。
- 多数 Git 操作 command 已存在并可复用。
- 项目中已有原生菜单（Tauri Menu）能力可复用到分支场景。

本变更是交互层重排，不是 Git 底层能力重写。

## Goals / Non-Goals

**Goals:**

- 用“右键菜单”统一分支操作入口，移除每行常驻 checkout 按钮。
- 保证菜单顶部固定展示分支 tracking 摘要，`Checkout` 固定第一可执行项，其余按分组顺序输出。
- 根据分支类型（当前/本地/远程）动态决定动作可见与可用状态。
- 保持现有 Git command 协议与核心执行链路不变。

**Non-Goals:**

- 不引入新的 Git 后端 command 协议。
- 不改造提交历史、diff viewer、分支分组算法。
- 不在本次实现高级工作流（例如交互式 rebase 向导）。

## Decisions

### Decision 1: 使用分支动作注册表驱动菜单构建

- 决策
    - 设计 `BranchContextAction` 注册表，按固定顺序定义动作、分组、可见条件、禁用条件、执行器。
    - 菜单渲染从注册表动态生成，避免分散的 `if/else` 拼接。
- 选择理由
    - 顺序与分组可测试、可审计，满足“tracking 顶栏 + Checkout 第一可执行项”的强约束。
    - 新动作扩展只需新增注册项，减少回归范围。
- 备选方案
    - 方案 A：在组件 JSX 中硬编码菜单项。
        - 缺点：可维护性差，条件分支膨胀。
    - 方案 B：后端返回菜单定义。
        - 缺点：耦合过深，前端交互迭代成本高。

### Decision 2: 复用现有 Git command，禁止新增协议

- 决策
    - `checkout/create/rename/delete/merge/rebase/push/update` 均映射到现有命令或既有组合流程。
- 选择理由
    - 本次目标是 UI 交互升级，复用现有 command 能最大化降低风险。
    - 避免引入额外后端行为差异，缩短验证路径。
- 备选方案
    - 方案 A：新增一条“菜单动作统一 command”。
        - 缺点：抽象收益低但迁移成本高，且不符合最小改动原则。

### Decision 3: 当前分支与远程分支的操作安全策略前置到菜单层

- 决策
    - 当前分支禁用 `Checkout`、`删除`。
    - 远程分支仅显示/启用有效动作，不允许无语义操作入口。
- 选择理由
    - 将错误阻断前移到交互层，降低误操作概率。
    - 与用户心智一致，减少失败弹窗频率。
- 备选方案
    - 方案 A：全部操作都可点，失败时再由后端报错。
        - 缺点：噪声高、体验差、可恢复性差。

### Decision 4: 右键菜单与键盘可访问性并行交付

- 决策
    - 菜单支持键盘上下选择与 `Esc` 关闭。
    - 打开位置锚定鼠标点，避免遮挡当前分支主信息。
- 选择理由
    - 维持可访问性与操作可预测性，避免仅鼠标路径可用。
- 备选方案
    - 方案 A：仅鼠标支持。
        - 缺点：无障碍能力不足，测试覆盖不完整。

### Decision 5: “分支 vs 工作树差异”采用两段式按需加载

- 决策
    - 第一步仅拉取差异文件元信息（路径/状态），渲染文件列表。
    - 第二步在用户点击具体文件后，按文件路径拉取该文件完整 diff 内容并展示。
- 选择理由
    - 大变更量场景下，避免一次性传输和渲染海量 diff 文本造成 UI 卡死。
    - 将性能成本延迟到用户真正关心的文件，显著降低首屏等待与内存峰值。
- 备选方案
    - 方案 A：继续一次性拉取全量 diff。
        - 缺点：分支差异文件较多时会出现明显卡顿甚至页面无响应。

### Decision 6: 菜单最上层展示分支 tracking 摘要（只读）

- 决策
    - 在菜单动作组之前新增只读 tracking 行，格式为 `<local-branch> -> <upstream-branch>`。
    - 当本地分支无 upstream 时，显示 `<local-branch> -> (未设置上游分支)` 占位文案。
    - tracking 行不可点击，不参与键盘动作焦点循环。
- 选择理由
    - 右键打开即暴露分支跟踪关系，可直接支撑“二开拉上游”场景下的快速确认。
    - 保留 `Checkout` 的位置稳定性（第一可执行项），避免操作记忆被破坏。
- 备选方案
    - 方案 A：将 tracking 信息放到 tooltip 或二级弹窗。
        - 缺点：可见性差，需要额外交互；无法满足“弹窗最上层直读”的要求。

### Decision 7: “与 `<current>` 比较”采用 IDEA 风格双向提交集合视图

- 决策
    - 将 `compare-current` 定义为“分支提交集合比较”，不再以文件差异列表作为主视图。
    - 视图必须同时展示两个方向：
        - `target \ current`：存在于目标分支但不存在于当前分支的提交。
        - `current \ target`：存在于当前分支但不存在于目标分支的提交。
    - 选择任一提交时，右侧（或对应详情区）展示该提交文件变更与提交信息摘要。
    - 在提交详情区点击任一文件时，打开单文件 diff 预览弹窗；关闭弹窗后保持 compare 工作区选中态不变。
        - 即使目标分支已被选中，点击 compare 仍必须打开该双向提交视图，不允许 no-op。
- 选择理由
    - 用户心智来自 IDE（如 IDEA）中的 Compare Branches；核心是“提交集合差异”，不是“文件全集差异”。
    - 双向展示能直接回答“谁领先了哪些提交”与“双方各自独有提交”两个关键问题。
- 备选方案
    - 方案 A：branch-to-branch 文件差异列表。
        - 缺点：丢失提交语义与方向信息，无法快速判断分支历史关系。
    - 方案 B：保持现状，仅 setSelectedBranch。
        - 缺点：在常见场景下无可见反馈，用户感知为功能失效。

## Risks / Trade-offs

- [Risk] 菜单动作过多导致认知负担增加 -> Mitigation: 使用分组分隔与稳定排序，保留常用动作在前两组。
- [Risk] 分支类型判断错误导致动作误启用 -> Mitigation: 建立统一 `branchActionGuard`，并增加类型矩阵测试。
- [Risk] 入口迁移后用户短期不适应 -> Mitigation: 保留短期提示文案（右键可查看更多操作）并在 release notes 说明变更。
- [Risk] 菜单与现有快捷键冲突 -> Mitigation: 复用既有快捷键绑定，不在本次新增全局快捷键。
- [Risk] 按需加载导致“点击文件后有二次等待” -> Mitigation: 增加文件级 loading 态与错误提示，并缓存已加载文件 diff。
- [Risk] tracking 信息缺失或展示过期引用 -> Mitigation: 优先复用分支列表同源 upstream 数据，并在空值时强制显示“未设置上游分支”占位文案。
- [Risk] 双向提交集合在大仓库中返回过慢 -> Mitigation: 首次加载限制提交数量并支持分页/继续加载。
- [Risk] 提交方向展示不清导致误读 -> Mitigation: 视图固定展示 `target \ current` 与 `current \ target`，并在标题与列表头双重标注。

## Migration Plan

1. 第一步：引入动作注册表与菜单状态容器，但暂时保留旧 checkout 按钮（灰度开关）。
2. 第二步：接入右键菜单并完成 tracking 顶栏、动作映射、禁用规则与确认流。
3. 第三步：移除行尾 checkout 按钮，打开新交互默认开关。
4. 第四步：补齐单测与集成验证，覆盖菜单顺序、禁用态、关键动作回归。
5. 第五步：为“显示与工作树的差异”切换至“文件列表优先 + 单文件详情按需加载”。
6. 第六步：将 compare 视图升级为“IDEA 风格双向提交集合 + 提交详情”。
7. 回滚策略：若线上出现高频故障，可通过 feature flag 回切到旧按钮入口（保留一版周期）。

## Open Questions

- `更新 (Update)` 在非当前本地分支上是否需要支持“仅拉取跟踪信息”的只读模式，还是统一禁用并提示？
- 菜单中的 `Push` 是否需要在无上游分支时提供“选择远程并建立 tracking”二级流程（当前提案暂不展开）？
- compare 视图中，双向提交列表默认展示条数与分页策略（例如每侧 200 条）如何配置最合适？
