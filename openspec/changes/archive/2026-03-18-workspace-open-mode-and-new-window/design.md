## Context

当前工作区导入链路把“选择目录”直接等价为“加入当前实例并激活”，未暴露打开策略。  
对于需要并行处理多个项目的 macOS 用户，这会导致频繁上下文切换，且无法通过统一的产品入口完成“新开客户端窗口承载新项目”的操作。  
现有代码已具备路径启动基础能力（`open-paths` 事件 + pending paths），可复用为新窗口导入的承载机制。

## Goals / Non-Goals

**Goals:**
- 在“添加工作区”时提供显式策略分流：`加入当前窗口` / `新开窗口`。
- 提供 `CmdOrCtrl+Shift+N` 语义的新建窗口入口，降低并行项目操作成本。
- 避免同一路径被重复写入 workspace 列表，保证多实例场景下行为可预测。
- 保持现有默认行为兼容，避免对既有用户造成破坏性变更。

**Non-Goals:**
- 本次不实现 VSCode 级别的同进程多窗口完整架构重构。
- 本次不引入 workspace 存储模型大改（如数据库化或全局事务层）。
- 本次不改动 AI 会话引擎协议与线程模型。

## Decisions

### Decision 1: 引入 `OpenMode` 策略层而非继续硬编码“当前窗口”

- 方案：在前端导入入口抽象 `OpenMode = current-window | new-window`，菜单点击与侧栏按钮统一走该策略层。
- 原因：把“导入目录”与“承载位置”解耦，后续可继续扩展（例如按场景默认值或策略开关）。
- 备选：
  - 继续沿用单链路 + 新增独立“新开窗口导入”菜单：会造成入口分叉与维护重复，放弃。

### Decision 2: 新窗口先采用“新实例模式”，不直接上同进程多窗口

- 方案：新窗口模式通过系统启动新实例并传递路径，复用已有 `open-paths` 导入链路。
- 原因：改动范围可控、交付快、风险低，且符合用户“像 VSCode 一样快速新开”核心诉求。
- 备选：
  - 同进程多窗口（多个 webview + window-scoped state）：涉及菜单路由、事件分发、状态隔离、窗口生命周期全链路重构，当前阶段成本过高，放弃。

### Decision 3: 去重策略前置到导入入口

- 方案：导入路径前先按规范化路径匹配现有 workspace；命中则激活已有项，不新增记录。
- 原因：多实例/重复导入场景下可以立即消除重复污染，且可独立于存储层演进。
- 备选：
  - 仅后端拒绝重复：前端体验差（失败提示而非“直接切换”），且不能覆盖部分启动导入链路，放弃。

### Decision 4: 兼容性优先，每次导入显式询问打开模式

- 方案：每次“添加工作区”都询问 `加入当前窗口` / `新开窗口`，不记忆上次选择。
- 原因：避免额外“是否记住”二次弹窗，降低交互负担并保持决策透明。

## Risks / Trade-offs

- [Risk] 新实例并发写 workspace 列表可能出现竞态覆盖。  
  → Mitigation: 前端去重 + 导入后主动刷新；后续补充存储写入锁/原子写增强。

- [Risk] 平台差异导致“新窗口”启动参数不一致。  
  → Mitigation: 优先保证 macOS 主路径，其他平台先提供可恢复 fallback（回退当前窗口导入或错误提示）。

- [Risk] 菜单事件当前依赖 `"main"` 窗口标签，扩展时容易遗漏。  
  → Mitigation: 在本次范围内先补“新建窗口”最小链路；后续把菜单事件改为面向 focused window 的统一分发。

- [Risk] 增加交互选择可能影响快捷操作效率。  
  → Mitigation: 仅保留一次模式询问弹窗，并允许快捷键直达“新建窗口”。

## Migration Plan

1. 增加 `OpenMode` 交互（每次单次询问），不替换旧入口。
2. 上线新建窗口菜单与快捷键，打通新实例路径导入。
3. 接入路径去重逻辑，覆盖手动导入与 `open-paths` 导入。
4. 灰度观察重复 workspace 比例与失败率；异常时可回滚到仅 current-window。

**Rollback Strategy**
- 关闭 `new-window` 分流并隐藏新入口，保留原有 add workspace 单链路。
- 去重逻辑可独立保留（非破坏性）。

## Open Questions

- 新窗口是否必须“仅打开目标项目视图”，还是允许加载全量 workspace 列表后自动激活目标项？
- `CmdOrCtrl+Shift+N` 在 macOS 是否仅创建空窗口，还是应支持“最近路径快速打开”？
- 存储层是否在本期同步引入文件锁/原子写保障，还是放到后续稳定性迭代？
