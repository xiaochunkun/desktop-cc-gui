## Why

当前“新建工作树代理”流程默认基于当前 `HEAD` 创建分支，用户在弹窗内看不到、也无法显式选择基线分支。该隐式行为在多发布线并行（
`main`、`chore/bump-version-*`）时容易把新工作树建在错误基线上，直接导致 PR 范围异常、审查成本上升与回滚困难。

## 目标与边界

- 目标
    - 在创建工作树时允许用户显式选择 `Base Branch`（本地/远端分支）。
    - 将最终使用的基线 commit 明确展示在 UI 中，避免隐式 `HEAD`。
    - 在创建前执行校验（分支存在、工作树路径可用、分支名合法）。
    - 支持创建成功后可选执行 `git push -u origin <branch>`，自动建立远端 tracking。
    - 统一工作区卡片头部的 `...` 与 `+` 入口，收敛为单一 `+` 入口并分组展示动作。
- 边界
    - 仅覆盖“创建工作树代理”流程，不扩展到完整 Git Flow 编排。
    - 仅覆盖“创建后一步发布到 `origin` + tracking 绑定”，不扩展到 PR 创建编排。
    - 不改变已有工作树的数据结构与存储位置。
    - `...` 中已有动作保持能力不变，仅迁移入口与重排信息架构。

## 非目标

- 不在本次引入自动 rebase、自动 cherry-pick、自动同步上游。
- 不在本次引入多远端策略选择（仅支持 `origin` 作为发布目标）。
- 不在本次实现跨仓库模板化分支策略（如组织级策略中心）。
- 不改动已有 PR 提交流程文档以外的协作制度。

## What Changes

- 在“新建工作树代理”弹窗新增 `Base Branch` 选择器，采用**仅下拉可选**（不允许自由输入），并按分组展示（`local/*`、`origin/*`、
  `upstream/*`）。
- 新增“基线预览”区域，显示将用于创建的 commit short SHA 与分支名。
- 创建动作从“默认当前 `HEAD`”改为“显式选择基线分支（默认占位为 `请选择`，必须由用户明确选择）”。
- 在创建弹窗新增“创建后推送到 `origin` 并设置 tracking”开关（默认开启，可关闭）。
- 在弹窗内为关键元素补充“新手可读说明 + 示例”，降低 Git 新手理解门槛（字段含义、典型值、按钮行为）。
- 对创建弹窗进行 UI 重构：宽版双栏布局（左侧引导、右侧表单），提升信息密度与可读性。
- 关键控件采用定制样式（自定义 select 外观、自定义 switch、强调态按钮），避免浏览器默认样式带来的视觉割裂。
- 当开关开启且创建成功后，执行 `git push -u origin <new-branch>`，并在 UI 中显示发布结果与跟踪分支信息。
- 增加创建前校验与错误提示：
    - 基线分支不可解析时阻断创建。
    - 目标分支名非法时阻断创建。
    - 目标工作树目录冲突时阻断创建。
- 增加创建后发布错误处理：
    - 认证或权限失败时提示可重试命令。
    - push 失败不回滚已创建本地工作树，改为显式提示后续补救步骤。
- 为该流程补充测试：
    - UI 选择与默认值测试。
    - 后端参数透传与校验测试。
    - 创建后 `push -u` 成功/失败路径测试。
    - 回归测试：未选择基线时不允许 silent fallback 到隐式 `HEAD`。
- 工作区卡片交互调整：
    - 将 `...` 和 `+` 合并成单一 `+` 按钮作为统一入口。
    - 点击 `+` 后展示分组菜单，第一组为“新建会话”：
        - `Claude Code`
        - `Codex`
        - `OpenCode`
        - `Gemini`
    - 第二组为“工作区操作”，承接当前 `...` 菜单中已有动作（保持行为不变）。
    - 两组动作均补齐小图标，组间有明确分隔线与组标题。

## 技术方案对比

### 方案 A：保持现状（隐式使用当前 HEAD）

- 优点：实现成本最低。
- 缺点：误建基线风险持续存在；问题不可见；用户心智负担高。
- 结论：不采纳。

### 方案 B：新增 Base Branch 选择器 + 显式 commit 预览（本提案）

- 优点：行为可见、可控；从源头降低错误基线 PR；与 Git 专业工具体验一致。
- 缺点：UI 与参数链路有一定改造成本。
- 结论：采纳。

### 方案 C：仅增加确认弹窗（显示“将基于当前 HEAD 创建”）

- 优点：改造较小。
- 缺点：仍然不能切换基线；只能“发现问题”，不能“避免问题”。
- 结论：不采纳。

## Capabilities

### New Capabilities

- `git-worktree-base-selection`: 在创建工作树时显式选择基线分支并可视化基线 commit。

### Modified Capabilities

- `git-branch-management`: 补充“创建相关操作必须显式基线/来源”的需求约束，禁止默认隐式 `HEAD`。

## Impact

- 前端
    - 工作树创建弹窗：新增基线分支下拉选择（仅选项可选）、分组展示、commit 预览与校验提示。
    - 新手说明层：为分支名、基线分支、基线预览、发布开关、设置脚本、按钮行为提供示例解释。
    - 布局层：桌面端采用 `guide rail + form` 双栏，移动端回退为上下堆叠，保持可用性。
    - 创建流程结果区：展示是否已发布到 `origin` 与 tracking 绑定结果。
    - Git 分支数据加载层：提供可用于工作树创建的候选分支集合。
    - 工作区卡片头部操作区：`... +` 双按钮改为单 `+`，并支持“新建会话组 + 工作区操作组”的分组菜单。
- 后端
    - 工作树创建命令入参：新增/强化 `baseRef` 参数约束与错误码。
    - Git 调用层：创建前解析 `baseRef` 到 commit，失败时返回可读错误。
    - 创建后发布逻辑：在开关开启时执行 `git push -u` 并返回可读发布状态。
- 测试
    - 前端组件单测新增基线选择与默认策略覆盖。
    - 后端命令测试新增 `baseRef` 解析失败/成功路径与 `push -u` 结果映射。

## 验收标准

- 用户在“新建工作树代理”弹窗中只能通过下拉选择基线分支，默认展示“请选择”。
- 创建前界面明确展示“分支名 + commit short SHA”的基线预览。
- 弹窗关键元素均有示例化解释，Git 新手可直接理解每个输入项与开关含义。
- 桌面端弹窗在不牺牲可读性的前提下显著加宽，并将引导说明固定在左侧区域。
- 当 `baseRef` 无效时，创建被阻止并显示可读错误。
- 不再出现“未显式选择基线仍默默按当前 HEAD 创建”的行为。
- 当“创建后推送到 `origin`”开关开启时，新分支创建成功后可自动完成远端推送并建立 tracking。
- 当推送失败时，界面明确反馈失败原因与下一步补救命令，本地创建结果保持可用。
- 新增测试通过，且不破坏现有 Git 历史/分支能力。
- 工作区卡片头部不再同时出现 `...` 与 `+`，仅保留单一 `+` 按钮。
- 点击 `+` 后优先展示“新建会话”分组，包含 `Claude Code/Codex/OpenCode/Gemini` 四个入口并带对应 icon。
- 菜单第二组完整保留原 `...` 中动作，且每个动作有对应 icon。
