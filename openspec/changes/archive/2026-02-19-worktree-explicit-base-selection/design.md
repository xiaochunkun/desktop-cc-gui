# Worktree Explicit Base Selection - Design

## Context

当前“新建工作树代理”流程存在一个核心问题：创建动作隐式依赖当前 `HEAD`，用户在 UI 中看不到真实基线，也无法在创建前切换到目标发布线分支。这会在多基线并行（如
`upstream/main` 与 `upstream/chore/bump-version-*`）时引入高概率误操作，最终表现为 PR 范围异常和审阅链路变长。并且在“二开”场景中，创建后还需手动执行
`git push -u origin <branch>`，遗漏后会影响协作和后续 PR 发起。

约束条件：

- 当前 Git 面板已有分支列表与分组能力（local/origin/upstream）。
- 创建流程已具备“分支名输入 + setup script”能力，但缺少基线选择参数。
- 需要保持现有工作树存储结构与命令接口兼容，避免破坏历史行为。

## Goals / Non-Goals

**Goals:**

- 在创建工作树弹窗中提供显式 `Base Branch` 选择，并展示最终解析到的 commit。
- 创建前进行基线、分支名、目标路径三类校验，阻止 silent fallback。
- 将前后端契约升级为“必须带 `baseRef`”，并保留可读错误映射。
- 提供创建后可选自动发布到 `origin` 并绑定 tracking 的一跳能力。
- 将工作区卡片头部的 `...` 与 `+` 合并为单一 `+` 入口，并按“新建会话/工作区操作”双分组组织菜单。

**Non-Goals:**

- 不引入自动 rebase / auto-sync / auto-cherry-pick。
- 不引入多远端智能路由（仅支持 `origin` 作为自动发布目标）。
- 不在本次改造分支策略管理（例如团队级基线策略中心）。
- 不重构工作树底层存储与目录规划。
- 不删除 `...` 菜单原有能力，仅调整入口形态与信息架构。

## Decisions

### Decision 1: UI 必须显式选择 Base Branch（仅下拉）

- 选型：在“新建工作树代理”弹窗中新增 `Base Branch` dropdown select（仅可从候选项选择，不允许自由输入）。
- 备选：继续隐式使用当前 `HEAD`，仅在确认弹窗提示。
- 取舍：选择“仅下拉”是为了彻底避免拼写错误、不可解析 ref 与隐式行为。

### Decision 2: 默认值策略采用“强制显式选择”

- 选型：基线分支默认值为占位项 `请选择`，用户必须手动确认目标基线后才可创建。
- 备选：自动预选 `upstream/main` 或当前分支。
- 取舍：自动预选仍可能被“无感确认”误用；强制显式选择更符合高风险操作的防呆原则。

### Decision 3: 后端创建命令必须接收 `baseRef` 并先解析

- 选型：创建前执行 `baseRef -> commit` 解析，解析失败立即返回结构化错误。
- 备选：前端先解析后端直接创建。
- 取舍：后端校验是最终防线，可避免 UI 与 CLI/其他入口行为不一致。

### Decision 4: 统一错误模型，禁止静默降级到 `HEAD`

- 选型：当 `baseRef` 缺失/无效时返回 `VALIDATION_ERROR`，前端阻断提交并展示可读提示。
- 备选：失败后自动 fallback 到 `HEAD`。
- 取舍：fallback 会再次引入根因，必须禁止。

### Decision 5: 创建后发布采用“显式开关 + 默认开启”

- 选型：在创建弹窗增加“创建后推送到 `origin` 并设置 tracking”开关，默认开启，可由用户关闭。
- 备选：强制自动推送、或完全不提供自动推送。
- 取舍：强制自动推送会污染远端实验分支；完全不支持会保留人工遗漏。显式开关在效率和可控性之间平衡更好。

### Decision 6: Push 失败不回滚本地 worktree，改为可恢复提示

- 选型：若 `git push -u origin <branch>` 失败，保留已创建本地工作树并返回结构化错误（含建议命令）。
- 备选：失败即回滚删除本地分支与工作树。
- 取舍：自动回滚风险高，且可能误删用户本地改动；保留本地并提示补救更安全。

### Decision 7: 为 Git 新手增加“元素级解释 + 示例”

- 选型：在弹窗中为每个关键元素增加简明解释和示例（分支名、基线分支、基线预览、推送开关、脚本、按钮行为）。
- 备选：仅保留字段名称与 placeholder。
- 取舍：本功能面向多角色开发者，增加示例化说明可显著降低新手误操作率和理解成本。

### Decision 8: 弹窗采用“左引导 + 右表单”的宽版双栏架构

- 选型：桌面端使用宽版双栏（左侧 Guide Rail、右侧 Form Panel）；移动端自动堆叠。
- 备选：继续单栏纵向流布局。
- 取舍：双栏更适合高信息密度场景，用户可在填写时持续看到引导，减少上下滚动与认知切换。

### Decision 9: 关键控件采用定制化样式实现

- 选型：对 Base Branch select、publish switch、主按钮做视觉重构（统一边框、层次、状态反馈）。
- 备选：沿用浏览器默认 select/checkbox。
- 取舍：定制化控件可提升整体一致性与品牌感，降低“系统默认控件”造成的视觉断层。

### Decision 10: 工作区头部动作入口收敛为单一 `+`

- 选型：将 `...` 与 `+` 合并为单一 `+` 按钮，点击后打开统一动作菜单。
- 备选：继续保留双入口（`...` 负责管理动作、`+` 负责新建会话）。
- 取舍：双入口在窄侧栏下易造成视觉噪声与心智分裂；单入口能降低认知成本并提升发现一致性。

### Decision 11: 菜单按“新建会话组 -> 工作区操作组”分层

- 选型：第一组固定为“新建会话”，包含 `Claude Code/Codex/OpenCode/Gemini`；第二组承接 `...` 现有动作。
- 备选：扁平化混排所有动作。
- 取舍：分组可将高频创建动作前置，同时保留管理动作的完整性，不破坏原有能力。

### Decision 12: 分组菜单的每个动作都配套图标

- 选型：为四个会话入口和原 `...` 动作都配置语义图标，组间使用分隔线与标题。
- 备选：仅保留文字菜单。
- 取舍：图标能强化动作语义，减少误点；在侧栏紧凑布局中更利于快速扫描与识别。

## Risks / Trade-offs

- [Risk] 远端分支列表较大导致弹窗加载变慢 -> Mitigation: 首次渲染仅显示最近使用 + 懒加载完整列表。
- [Risk] 强制显式选择增加一次点击 -> Mitigation: 下拉分组 + 示例说明 + 创建按钮前预览，降低认知摩擦。
- [Risk] 宽版双栏在窄屏可能拥挤 -> Mitigation: `max-width` + 响应式断点，在移动端回退为单栏堆叠。
- [Risk] 后端参数升级影响旧调用路径 -> Mitigation: 对旧调用路径先加兼容错误提示并在调用方统一补参。
- [Risk] 测试覆盖不足导致回归 -> Mitigation: 补 UI 单测 + 后端命令测试 + E2E 回归场景。
- [Risk] 远端鉴权不足导致自动 push 失败 -> Mitigation: 返回可读错误 + 展示手工 `git push -u` 补救命令。
- [Risk] 默认开启自动 push 可能创建过多远端分支 -> Mitigation: 提供显式开关并记录用户最近选择偏好。
- [Risk] 单一 `+` 入口可能让用户误以为“工作区操作”被删除 -> Mitigation: 在菜单中保留第二组并按原语义命名，组标题明确标识。
- [Risk] 新增 `Gemini` 入口在未配置能力时不可用 -> Mitigation: 保留入口但显示禁用态与可读原因提示。

## Migration Plan

1. 前端阶段

- 新增 `Base Branch` 下拉字段与候选分支数据源（默认占位 `请选择`）。
- 新增基线预览（branch + short SHA）。
- 新增新手说明文案（字段含义 + 示例）。
- 合并工作区头部 `... +` 按钮为单 `+`，接入两组式菜单结构与 icon 渲染。

2. 后端阶段

- 扩展创建工作树命令入参与校验逻辑，强制 `baseRef`。
- 增加错误类型与 message key。
- 增加创建后可选 `push -u origin` 的执行链路与错误映射。

3. 测试阶段

- 增加“必须显式基线”的回归测试，验证不再 fallback。
- 增加“无效 baseRef / 非法分支名 / 路径冲突”测试。
- 增加“自动 push 成功 / 权限失败 / 网络失败”测试。

4. 发布阶段

- 小流量验证：在开发分支启用并收集创建失败率与失败原因。
- 监控自动 push 成功率与失败类型分布（鉴权、网络、冲突）。
- 全量启用：确认失败率可控后全量发布。

5. 回滚策略

- 若出现高频阻断且影响开发效率，可临时回滚到旧 UI，但保留后端错误日志。
- 回滚时必须保留显式告警，避免完全恢复隐式行为而再次引发误建。

## Open Questions

- 默认推荐基线是否需要按仓库级配置覆盖（例如某仓库固定 `upstream/release/*`）？
- Base Branch 选择器是否需要提供“最近使用”置顶策略？
- 是否需要在创建完成后记录 `baseRef` 到 worktree 元数据，便于后续审计？
- `Gemini` 入口是否在首版默认可用，还是首版以禁用态占位并引导配置？
