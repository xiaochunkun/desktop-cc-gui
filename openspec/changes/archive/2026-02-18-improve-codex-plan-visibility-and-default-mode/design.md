## Context

现有 Plan 面板逻辑把“模式引导”优先级放在“已有数据展示”之前，导致 Code 模式下即使线程已存在计划，也可能被“切换到 Plan
模式后可查看计划”的空态替代。与此同时，批量编辑卡片中的 Plan 快览受限于容器高度与 overflow，长计划会被截断。右侧 Plan
面板缺少显式关闭操作，用户只能依赖全局布局变更绕开。默认协作模式仍为 Code，也会放大首轮“看不到计划”的概率。

## Decisions

### Decision 1: 数据可见性优先于模式提示

- 规则：`hasPlanData === true` 时始终渲染计划内容，模式仅影响提示文案，不影响已有数据显示。
- 结果：Code 模式不再成为计划阅读阻断条件。

### Decision 2: 快览层与卡片层分离滚动策略

- 批量编辑卡片中的 Plan 快览改为“固定头 + 可滚动内容区”，避免父容器裁切。
- 在移动与窄宽场景下，快览最大高度采用 viewport 比例上限并保留滚动。

### Decision 3: 右侧 Plan 面板提供显式关闭

- 在 Plan 面板标题区新增 close action。
- 关闭仅影响面板可见状态，不清空计划数据。
- 保留“查看完整 Plan 面板”入口用于恢复。

### Decision 4: Codex 默认协作模式改为 Plan

- 仅对 Codex 生效。
- 优先级：用户显式选择 > 持久化值 > 默认值（Plan）。

### Decision 5: Codex 实时活动与详情卡片合并摘要策略

- 实时活动标签与详情卡片共享“摘要拼装规则”。
- 有 tool 执行事实（尤其 `commandExecution`）时，实时活动优先展示命令摘要，reasoning 作为补充上下文。
- 目标是让用户在“实时流”和“详情区”看到的是同一事实的不同层级，而不是两套互相偏离的文案。
- 合并策略仅在 `activeEngine === "codex"` 时启用，Claude/OpenCode 路径保持原样。

## Risks / Trade-offs

- 默认切 Plan 可能改变部分用户首轮行为预期。
    - 缓解：保留快速切回 Code 的显式入口，且记住用户选择。
- 快览区域可滚动后，若样式不当会出现双滚动。
    - 缓解：统一滚动容器边界并限制父级 overflow。
- 实时活动合并后可能导致文案过长。
    - 缓解：采用“主摘要 + 次摘要”两层展示，并对主摘要做长度上限与省略。

## Verification

- 单元/组件测试：
    - Code 模式 + 已有计划数据 => 计划可见。
    - 批量编辑卡片长计划 => 可滚动看到最后一条。
    - Plan 面板 close 点击 => 面板隐藏且可再次打开。
    - Codex 默认模式初始化 => Plan。
    - Codex 实时活动与详情卡片摘要规则一致，且仅 Codex 生效。
- 手工回归：
    - 在含批量编辑卡片的历史会话中验证 Plan 快览无截断。
