## Context

当前文件引用进入 Composer 的主路径是：

- 文件树节点 `+` 按钮通过 `window.handleFilePathFromJava(...)` 注入 `@path` 文本；
- ChatInputBox 通过既有 `pathMappingRef + renderFileTags` 链路完成标签渲染与状态同步。

现状问题是交互入口单一，无法直接满足“拖拽文件树节点/拖拽系统外部文件或文件夹到输入框”的习惯操作。与此同时，系统已经具备两类可复用基础能力：

- DOM 层拖拽接收（`usePasteAndDrop`，目前偏文本/图片）；
- Tauri 窗口级拖拽事件桥（`subscribeWindowDragDrop`，可拿到外部路径列表）。

本变更属于跨模块 UI 交互扩展（文件树 + Composer + dragDrop 服务），但不涉及协议、后端存储或执行链路改造。
现有文件树“单击即打开/展开”语义会吞掉稳定选择态，导致多选拖拽无法自然成立，因此需要先调整交互模型再接入批量拖拽。

## Goals / Non-Goals

**Goals:**

- 为文件树节点提供拖拽源能力，支持将文件/文件夹路径拖到 Composer。
- 为 Composer 提供统一拖拽接收入口，同时覆盖应用内拖拽与系统外部拖拽。
- 拖入后严格复用现有 `@file` 引用注入与渲染链路，保证发送行为零回退。
- 在无效 payload、非法路径、非目标区域 drop 场景下保持可恢复与不崩溃。
- 明确拖拽处理边界，确保新增交互仅在 Composer 输入区生效，不向其他面板扩散。
- 提供 Win/macOS 路径兼容保护，确保跨平台拖拽行为一致可预期。
- 将文件树交互调整为“单击选中，双击打开/展开”，为多选拖拽提供统一前置语义。
- 提供 Win/mac 兼容多选手势映射，保证选择行为与平台习惯一致。

**Non-Goals:**

- 不新增二进制文件上传能力（除既有图片附件能力外）。
- 不新增目录递归展开或批量文件自动注入策略。
- 不修改消息提交 payload 结构、模型调用契约和引用格式定义。

## Decisions

### Decision 1: 采用“双通道拖拽输入”，统一汇聚到单一注入函数

- 选择：
- 通道 A：DOM `onDrop/dataTransfer` 处理应用内拖拽与可解析文本；
- 通道 B：Tauri `onDragDropEvent` 处理外部文件系统路径。
- 理由：
- DOM 通道对外部路径稳定性有限，单用 DOM 无法可靠覆盖“项目外文件夹拖入”；
- Tauri 通道已经在项目中存在并可稳定提供 `paths`，能补齐外部拖拽。
- 备选方案（未采纳）：
- 仅 DOM：实现最小但覆盖不足；
- 仅 Tauri：会丢失纯 Web 拖拽语义，且内部拖拽适配成本更高。

### Decision 2: 文件树拖拽数据只传“绝对路径列表”，不传文件内容

- 选择：
- 文件树节点设置 `draggable`，`dragstart` 写入标准文本载荷（绝对路径，支持多路径预留）。
- 理由：
- 本需求目标是“引用路径”，不是“上传内容”；
- 与 `+` 路径注入语义一致，最小化行为差异。
- 备选方案（未采纳）：
- 传递自定义 JSON payload（含更多元信息）：扩展性更强但当前无必要，增加兼容负担。

### Decision 2.5: 文件树交互语义改为“选择优先”，双击触发打开/展开

- 选择：
- 单击行：仅更新选中态（active selection）。
- 双击文件：触发打开文件（复用现有 Tab 打开契约）。
- 双击文件夹：触发展开/折叠。
- 理由：
- 单击必须让位于“选择”才能承载多选拖拽；
- 双击触发打开符合桌面文件管理习惯，心智成本更低。
- 备选方案（未采纳）：
- 保持单击打开并引入额外选择控件：可行但 UI 与交互复杂度明显上升。

### Decision 3: 复用既有 `handleFilePathFromJava` 等价逻辑，避免双实现漂移

- 选择：
- 在 ChatInputBox 内抽出“路径列表注入”统一方法；`+` 按钮与拖拽都调用同一套逻辑（校验、mapping、文本插入、标签渲染、状态同步）。
- 理由：
- “拖入后逻辑不要变”是硬约束，复用可确保行为同源；
- 降低未来维护时的回归风险。
- 备选方案（未采纳）：
- 为拖拽单独实现一套插入逻辑：短期可行，但极易出现格式/状态不一致。

### Decision 4: 保留现有安全校验策略并扩展到拖拽入口

- 选择：
- 所有拖拽路径先经过现有 `validateFilePath`；
- 无效路径直接忽略（可选 toast），不阻断其他有效路径插入。
- 理由：
- 现有能力已覆盖路径遍历与敏感路径拦截；
- 保持“安全失败 + 输入框可用”契约。
- 备选方案（未采纳）：
- 拖拽入口绕过校验：存在明显安全与稳定风险。

### Decision 5: 交互边界收敛到 Composer 输入区，避免能力扩散

- 选择：
- 仅当 drop 命中 Composer 输入区时消费路径 payload；非目标区域一律不注入引用。
- 文件夹 payload 仅作单条路径引用，不在拖拽链路引入目录扫描或递归展开。
- 理由：
- 防止拖拽副作用污染消息列表、文件预览、侧边文件树等区域行为。
- 将需求严格限制在“引用入口增强”，避免范围膨胀导致回归面失控。
- 备选方案（未采纳）：
- 全局 drop 捕获并自动注入：实现便捷但边界不可控、误触发风险高。

### Decision 6: Win/macOS 兼容保护采用统一路径归一化策略

- 选择：
- 对拖拽路径统一做分隔符归一化后再进入引用注入逻辑。
- Windows 路径匹配采取盘符大小写不敏感策略；macOS 保持原路径显示并兼容 Finder 拖拽常见 payload。
- 理由：
- 外部拖拽 payload 在 Win/macOS 存在天然差异，不统一归一化会产生“同一路径重复/失配”问题。
- 保留平台差异最小必要处理，避免平台专有分支扩散到上层业务。
- 备选方案（未采纳）：
- 完全按原始字符串处理：实现简单但跨平台一致性不足，回归风险高。

### Decision 7: Win/mac 多选手势采用平台原生映射

- 选择：
- macOS：`⌘+Click` 切换选择项，`Shift+Click` 区间选择。
- Windows：`Ctrl+Click` 切换选择项，`Shift+Click` 区间选择。
- 拖拽起点命中已选集合成员时，携带整组已选路径；否则按单项拖拽处理。
- 理由：
- 保持平台一致性，减少学习成本；
- 使多选拖拽和单项拖拽行为都具可预测性。
- 备选方案（未采纳）：
- 自定义非平台手势：一致性弱，易与用户既有操作习惯冲突。

### Decision 8: 内部拖拽 bridge 生命周期由拖拽源收口，避免跨层抢先清理

- 选择：
- `FileTreePanel` 拖拽源在 `dragstart` 建立 bridge，在 `dragend/drop` 统一清理；
- Composer 侧仅在成功消费或明确无效时清理，不在捕获阶段 `document dragend` 抢先清理。
- 理由：
- 实际运行中存在“Composer 捕获层先于文件树源层执行 dragend”的时序，抢先清理会导致内部拖拽路径丢失；
- 生命周期收口到拖拽源可避免跨层 race condition，保证内部拖拽稳定可插入。
- 备选方案（未采纳）：
- 双方都清理并依赖时间窗兜底：实现复杂且仍存在偶发空路径风险。

### Decision 9: 拖拽遮罩采用“命中优先 + 单输入框视觉兜底”

- 选择：
- 命中到 Composer 输入区时显示遮罩并进入可 drop 态；
- 命中不稳定但页面仅有单个 Composer 时，显示视觉遮罩兜底，不改变 drop 判定边界。
- 理由：
- 在部分运行时（跨 pane/native drag）命中事件不稳定，直接导致“可拖入但无反馈”；
- 视觉兜底可降低交互不确定性，同时保持功能边界不扩散。
- 备选方案（未采纳）：
- 全局常亮遮罩：误导性强，且会放大非目标区域误操作感知。

## Risks / Trade-offs

- [双通道事件重复触发导致重复插入] → 通过路径去重（按规范化绝对路径）与短时幂等窗口抑制重复消费。
- [文件夹拖拽在不同平台 payload 差异] → 优先消费 Tauri `paths`，DOM 文本作为兜底；回归覆盖 macOS/Windows 常见场景。
- [与既有图片拖拽能力冲突] → 明确优先级：图片附件链路优先；非图片路径引用走文件引用链路；互斥分支加单测。
- [输入区命中判断误差导致非目标区域也触发] → 使用输入框 bounding rect 命中判断，drop 时再次校验。
- [Win/macOS 路径格式差异导致重复或失配] → 建立统一归一化函数并加入平台样例测试（Windows 盘符/反斜杠、macOS Finder 路径）。
- [单击改选中导致“打开”路径感知回退] → 在 UI 上提供清晰选中态与双击提示，并通过回归测试锁定双击打开契约。
- [多选语义与右键菜单/键盘导航冲突] → 明确优先级：右键不破坏已选集合，键盘方向键仅移动主选中项。
- [拖拽事件时序 race 导致内部路径丢失] → 由拖拽源统一收口 bridge 生命周期，并补充 dragend 回归用例。
- [拖拽命中事件不稳定导致缺少遮罩反馈] → 增加单输入框视觉兜底策略，且不放宽实际 drop 边界。

## Migration Plan

- 前端增量发布，无后端 schema 或数据迁移。
- 分步上线建议：
- 先合并统一注入函数与测试；
- 再接入文件树拖拽源；
- 最后接入外部拖拽路径并完成跨平台验证。
- 最后执行 Win/macOS 兼容矩阵验收并固化回归用例。
- 最后执行“单击/双击/多选/拖拽”交互矩阵验收并固化快照与行为测试。
- 回滚策略：
- 若出现拖拽回归，可通过 feature flag/条件分支快速关闭新拖拽入口，保留 `+` 与手动 `@` 引用路径。

## Open Questions

- 外部拖拽文件夹时，是否需要在 UI 上增加“目录引用”视觉标识（当前可沿用普通文件引用样式）？
- 多路径同时拖入是否需要上限（例如 50 条）以避免输入区性能抖动？
- 双击判定窗口（如 250ms / 300ms）是否采用全局配置，避免不同平台体感不一致？
