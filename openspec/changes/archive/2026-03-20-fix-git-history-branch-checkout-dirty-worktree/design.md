## Context

当前 Git History 分支签出链路在“工作区原本 clean”的情况下，仍可能在签出后留下大量 `(A)/(U)/(M)` 差异文件。现象表现为：

- 当前分支标识已更新到目标分支；
- 但工作树与暂存区出现非预期差异；
- 后续提交/变基/合并操作被污染或阻塞。

该问题属于高风险基础链路缺陷，覆盖本地分支与远端跟踪分支场景。设计目标是在不改变用户交互入口的前提下，恢复 deterministic checkout 语义：成功签出后，仓库状态与目标分支一致且保持 clean。

## Goals / Non-Goals

**Goals:**
- 保证 clean precondition 下签出成功后的 clean postcondition。
- 统一本地与远端跟踪分支签出语义，减少执行分叉。
- 为签出失败提供可诊断、可恢复的错误反馈，避免“伪成功”状态。
- 增加覆盖分支差异集较大场景的回归测试，防止再次回归。

**Non-Goals:**
- 不新增“stash and checkout / discard and checkout”的完整交互编排。
- 不调整 Git History 面板的菜单结构、视觉呈现或其他 Git 操作入口。
- 不改造 create/delete/merge/rebase/reset/push/pull 语义。

## Decisions

### Decision 1: 本地分支签出与远端签出统一到 Git CLI 语义

- 选择：将本地分支签出执行路径对齐到 Git CLI 语义（与现有远端跟踪分支路径一致）。
- 原因：
  - 与用户本地 Git 行为一致，可预期性更高；
  - 减少“本地/远端两套签出语义”导致的维护分叉；
  - 更易复现与排障，降低跨平台差异风险。
- 备选：
  - 保留 libgit2 本地签出，仅修补 checkout 选项与基线控制；
  - 取舍：可行但语义细节复杂、后续维护成本偏高，不作为本次首选。

### Decision 2: 增加签出后状态一致性校验

- 选择：签出成功返回前增加最小一致性验证（分支与工作树状态一致）。
- 原因：
  - 防止“分支指示更新但工作区异常”的伪成功；
  - 将异常尽早暴露给调用方，避免脏状态扩散到后续操作。
- 备选：
  - 仅依赖底层命令返回码，不做后验检查；
  - 取舍：实现更简单，但无法覆盖部分边缘异常，不满足本次可靠性目标。

### Decision 3: 失败路径保持原子语义与可恢复提示

- 选择：签出失败必须保持“当前分支状态不变 + 工作区不新增副作用差异”，并输出可读错误提示。
- 原因：
  - 保证用户可立即重试或改走命令行恢复；
  - 避免 UI 层出现误导性的“切换成功”反馈。
- 补充：dirty checkout 在本次保持“前置阻断 + 引导提示”策略，不扩展 stash/discard 向导交互。
- 备选：
  - 失败后仅抛原始错误，忽略 UI 语义一致性；
  - 取舍：无法满足前端稳定反馈要求。

### Decision 4: 回归测试覆盖“分支文件集差异大”的高风险场景

- 选择：新增后端与前端最小回归测试，覆盖 A/B 分支反复切换仍 clean。
- 原因：
  - 本问题易在大差异分支间复发，必须通过自动化测试锁定。
- 备选：
  - 仅手工验证；
  - 取舍：不可持续，回归风险高。

## Risks / Trade-offs

- [Risk] Git CLI 在不同平台的错误文案存在差异  
  → Mitigation：沿用现有错误归一化映射，测试断言以错误类别而非完整文案为主。

- [Risk] 签出后增加状态校验会带来轻微额外开销  
  → Mitigation：仅在 checkout 命令链路执行一次最小校验，避免高频轮询。

- [Risk] 改动签出核心路径可能影响既有分支流程  
  → Mitigation：保留最小回滚路径（revert checkout 改动提交）并在回归通过后再合入。

## Migration Plan

1. 在后端引入统一签出执行路径（本地/远端跟踪分支语义对齐）。
2. 增加签出后的一致性校验与错误映射，保证失败不伪成功。
3. 前端保留原交互入口，仅增强失败反馈处理与状态刷新时机。
4. 增补后端与前端回归测试，覆盖 clean A/B 往返切换场景。
5. 合入前执行最小门禁；若出现回归，通过提交级 revert 快速回退。

## Open Questions

- `Stash and Checkout / Discard and Checkout` 是否应在后续独立 change 中实现为完整向导？
- 是否需要将“签出后状态一致性校验”抽象为通用 guard，复用于 reset/rebase 等写操作？
- 错误提示中是否要显式附带“一键恢复建议”（例如建议执行的 git 命令）？
