## Why

当前 Spec Hub 入口与会话切换存在多处影响日常使用的问题：

- 打开 Spec Hub 面板后，从左侧点击会话无法切换到目标会话，Spec Hub 仍停留在前台，导致导航被阻断。
- Spec Hub 入口仅位于项目欢迎页，进入会话后缺少稳定入口，无法在工作流中快速返回。
- OpenSpec 变更创建成功后，收尾反馈弹窗将“变更文件”显示为`(无)`，与实时输出和实际产物不一致，影响结果可信度判断。
- 变更列表中的分组折叠能力仅覆盖“已归档”视图；在“全部”与其它视图下仍是长列表，不利于按日期批次快速定位。
- 图 1（`pasted-image-1`）中的按钮组前漏斗 icon 与当前信息结构不匹配，用户更需要“一键展开/折叠”控制。

结合截图（`pasted-image-1`、`pasted-image-2`）可见，Spec Hub 为高频操作面板，需要在所有列表视图保持一致的信息折叠体验与更清晰的视觉锚点。

## 目标与边界

- 目标：修复“会话点击后无法退出 Spec Hub”问题，确保会话切换可达。
- 目标：在左侧侧边栏底部新增 Spec Hub icon 入口，支持任意页面快速进入。
- 目标：修复收尾反馈弹窗的“变更文件列表”误判为空问题，确保创建成功时准确展示文件变更列表。
- 目标：将按日期分组的树形折叠能力扩展到变更列表“全部”视图，并与“已归档”保持一致交互模型。
- 目标：将按钮组前漏斗 icon 替换为“展开全部/折叠全部”控制入口，降低长列表操作成本。
- 目标：在分组节点与变更条目中增加少量语义化 icon 点缀，提升扫描效率且不干扰信息密度。
- 目标：保持现有欢迎页入口兼容，不破坏已有路径。
- 边界：不改动 OpenSpec CLI 协议与后端执行语义。
- 边界：不调整 Spec Hub 三栏信息架构。
- 边界：不修改已有归档目录命名规则，仅基于现有名称做前端分组渲染。

## Non-Goals

- 不在本提案中重做 Spec Hub 全量 IA。
- 不在本提案中新增 Spec Hub 权限模型。
- 不在本提案中替换欢迎页其它入口布局。

## What Changes

- 新增会话导航联动规则：当 Spec Hub 打开时，点击任意会话必须切换到该会话，并关闭/退出 Spec Hub 前台展示。
- 新增左侧 rail 底部 Spec Hub icon 入口规则：入口在 workspace 内保持稳定可见，点击后进入 Spec Hub。
- 明确入口一致性：欢迎页入口继续可用，侧栏入口提供随时可达能力。
- 新增反馈一致性规则：当 OpenSpec 变更创建成功且存在变更文件时，收尾反馈弹窗必须展示非空文件列表，不得错误显示`(无)`。
- 扩展分组范围规则：在“全部”与“已归档”视图中，将目录名符合日期前缀模式（`YYYY-MM-DD-*`）的变更按日期节点分组，并采用可展开/折叠树结构展示。
- 新增全局折叠控制规则：按钮组前提供“展开全部/折叠全部”按钮，替代漏斗 icon，用于批量展开或收起当前视图下所有日期分组节点。
- 新增 icon 点缀规则：日期分组节点、已验证/进行中等状态条目可使用语义化 icon 强化识别；icon 为视觉辅助，不替代文本状态表达。
- 新增兼容规则：不符合日期前缀的条目归入“其它”分组，避免数据丢失。

## Capabilities

### Modified Capabilities

- `spec-hub-workbench-ui`: 增加“会话点击优先导航并退出 Spec Hub 面板”交互契约。
- `spec-hub-workbench-ui`: 增加“收尾反馈弹窗准确展示变更文件列表”数据解析与展示契约。
- `spec-hub-workbench-ui`: 扩展“变更列表按日期前缀树形分组展示”到“全部+已归档”视图。
- `spec-hub-workbench-ui`: 增加“按钮组前展开/折叠全部控制 + 语义化 icon 点缀”视觉与交互契约。
- `workspace-sidebar-visual-harmony`: 增加“侧栏 rail 底部 Spec Hub icon 入口”布局与行为契约。

## 验收标准

1. 当 Spec Hub 面板处于打开状态时，点击左侧任意会话 SHALL 成功进入目标会话页面。
2. 会话切换成功后，Spec Hub SHALL 不再覆盖当前会话视图。
3. 左侧 rail 底部 SHALL 提供 Spec Hub icon 入口，并在会话页可触达。
4. 点击侧栏 Spec Hub icon SHALL 进入 Spec Hub，且不破坏原欢迎页入口。
5. 入口新增 SHALL 保持侧栏滚动与层级规则不回退。
6. 当变更创建成功且输出包含变更文件时，收尾反馈弹窗 SHALL 展示对应文件列表，不得显示`(无)`。
7. 在“全部”与“已归档”视图中，目录名匹配`YYYY-MM-DD-*`的条目 SHALL 按日期分组为可展开树节点，而非平铺列表。
8. 按钮组前 SHALL 提供“展开全部/折叠全部”控制，并替代原漏斗 icon 的交互职责。
9. 用户点击“展开全部/折叠全部”后，当前视图内所有日期分组节点 SHALL 同步切换到目标状态。
10. 每个日期节点展开后 SHALL 展示该日期下的变更子项，且子项点击行为与当前列表一致。
11. 不匹配日期前缀的条目 SHALL 可见且归入“其它”分组。
12. 分组节点与状态条目新增的 icon 点缀 SHALL 不影响可访问名称、文本状态和点击热区。

## Impact

- 影响范围：Spec Hub 前端导航交互、侧栏入口布局、变更列表信息架构与顶部控制区。
- 代码影响：前端 UI 事件联动、rail 入口注册、列表分组与树形渲染逻辑、展开/折叠批量控制与 icon 映射。
- 协作影响：降低 Spec Hub 与会话导航互斥造成的操作中断，提升长列表浏览效率与状态可读性。

## 追加范围（2026-02-26）：严格门禁前后对齐

### 背景

当前归档前置校验（Archive preflight）能够正确拦截 delta 语义错误，属于必须保留的强门禁能力；但问题在创建/追加提案阶段未被提前暴露，用户通常在“验证/归档前”才发现
MODIFIED 与目标 spec 不匹配，修复成本和认知成本偏高。

### 目标

- 保留并继续强化归档严格门禁，不降低拦截标准。
- 将同一规则前置到“提案创建/追加完成后”和“验证触发前”，实现前后判定一致。
- 在前置失败时提供可操作修复建议（例如 ADDED/MODIFIED 调整建议），避免用户盲改。

### 变更内容

- 新增“提案后 preflight 检查”：create/append proposal 成功后，自动执行 delta-target 兼容性检查并回显结果。
- 新增“验证前 preflight 拦截”：点击验证前先做同款 preflight；失败则阻止 strict validate 执行并展示修复指引。
- 统一规则实现：提案后、验证前、归档前三处复用同一 preflight 解析与判定逻辑，避免规则漂移。
- 新增错误可操作化：对 requirement 缺失场景提示“改 ADDED 或对齐目标同名 Requirement”。
- 保持 AI takeover 作为补救链路，但不替代门禁判定。

### 验收标准（追加）

1. 对同一 change，提案后/验证前/归档前三处 preflight blocker 集合一致。
2. 当 delta 使用 MODIFIED 且目标 spec 不存在同名 requirement 时，提案后即能看到 blocker，而不是仅在归档阶段暴露。
3. 前置检查失败时，验证动作 SHALL 被阻断并展示可操作修复建议。
4. 归档门禁规则保持不变，不引入宽松路径。
5. 外部 spec root（custom root）场景与工作区内置 root 场景判定一致。

## 追加范围（2026-02-26）：Spec Hub i18n 与硬编码清理

### 背景

当前 Spec Hub 模块仍存在局部写死文案与硬编码状态文案，导致中英文切换不完整、部分提示与按钮文本无法随 locale
变化，且后续维护时容易出现“新增文案未纳入 i18n”的回归。

### 目标

- 对 Spec Hub 模块进行一次面向 i18n 的整理，补齐遗漏文案的国际化接入。
- 清理 UI 层可见硬编码文案，统一收敛到 i18n key 管理。
- 保持当前交互行为不变，仅修复文案来源与多语言一致性问题。

### 变更内容

- 建立 Spec Hub 文案清单：覆盖按钮、分组标题、空态文案、提示文案、状态标签、反馈弹窗字段标题。
- 将可见硬编码文案替换为 i18n key（含“其它”、“展开全部/折叠全部”、“变更文件”、“(无)”等已知高频项）。
- 补齐缺失 locale 条目（至少 `zh-CN` 与 `en-US`），并保证 key 命名与模块边界一致（`specHub.*`）。
- 为新增与迁移文案补充最小回归验证，确保语言切换后 UI 文案同步刷新且无 key 泄漏。
- 明确治理边界：日志/debug 输出可保留英文常量，不纳入 UI i18n 强约束。

### 验收标准（追加）

1. Spec Hub 可见 UI 文案 SHALL 通过 i18n key 渲染，不得出现新增硬编码中文/英文文本。
2. 在 `zh-CN` 与 `en-US` 两种 locale 下，核心流程（列表浏览、创建反馈、验证前拦截提示）文案 SHALL 可读且语义一致。
3. 对“其它”“展开全部/折叠全部”“变更文件”“(无)”等关键文案 SHALL 完成 i18n 接入，且切换语言立即生效。
4. UI 中 SHALL 不展示原始 i18n key（例如 `specHub.xxx`）作为回退文本。
5. 本次整理 SHALL 不改变现有交互路径与权限语义，仅影响文案来源与显示一致性。
