## Context

当前右侧文件树直接渲染工作区一级条目，缺少显式根节点，导致层级语义与用户心智模型不一致。提案已确定新增单一工作区根节点，并在根节点复用现有目录右键菜单能力。该变更横跨前端树模型、交互事件分发、后端路径解析与现有回归链路（多 Tab、Diff/Preview、特殊目录渐进加载）。

约束：
- 必须兼容现有目录菜单命令链路，不新增分叉命令体系。
- 必须保持特殊目录渐进加载性能特征。
- 根节点路径必须严格绑定 active workspace root，禁止越界。
- 搜索输入框需上移至顶部工具行，并与计数/操作区共存且不遮挡。
- 根节点顶部区需保持 sticky，文件滚动仅发生在内容列表区。

## Goals / Non-Goals

**Goals:**
- 在文件树引入唯一工作区根节点，统一渲染层级（`root -> children`）。
- 根节点支持折叠/展开，默认展开，且恢复子树展开状态。
- 根节点右键菜单复用现有目录菜单动作与执行链路。
- 将文件搜索输入框并入顶部工具行，移除独立第二行布局。
- 根节点顶部区在滚动时保持吸顶固定，且操作按钮持续可交互。
- 保证现有多 Tab 打开与文件查看链路零回退。

**Non-Goals:**
- 不引入 multi-root workspace 模型。
- 不重写文件树搜索与索引。
- 不调整特殊目录白名单定义。

## Decisions

### Decision 1: 采用“真实根节点模型”而非“渲染层伪根”
- 选择：在文件树 domain model 中引入真实 root entity，API 与前端状态统一按 root 节点处理。
- 原因：避免数据语义与 UI 语义分裂，减少路径映射与状态持久化复杂度。
- 备选方案：仅前端渲染时注入虚拟根节点。
- 不选原因：后续上下文菜单、测试快照、展开状态恢复需要额外映射层，维护成本高。

### Decision 2: 根节点菜单复用现有目录 action handlers
- 选择：根节点上下文菜单事件直接复用目录菜单 command dispatch；根节点仅作为 path provider（workspace root path）。
- 原因：最大化复用现有稳定链路，降低行为偏差风险。
- 备选方案：为根节点单独实现一套 menu/action。
- 不选原因：会引入行为漂移与重复测试面。

### Decision 3: 特殊目录渐进加载语义保持不变，仅调整初始树包裹层
- 选择：初始 payload 变为“workspace root + direct children”，special subtree 的懒加载触发条件保持现有规则。
- 原因：确保性能与用户体感不退化。
- 备选方案：在根节点展开时一次性预取更多层。
- 不选原因：会放大大仓库加载开销，与已归档的渐进加载能力冲突。

### Decision 4: 根节点状态纳入现有 tree UI state store
- 选择：将 `rootExpanded` 与 root-level context state 纳入现有树状态存储（同一 state slice）。
- 原因：保证刷新、重渲染、工作区切换时行为一致，减少跨 store 同步。
- 备选方案：独立维护 root 专用状态。
- 不选原因：状态分散，调试复杂度上升。

### Decision 5: 搜索框并入顶部工具行并采用弹性布局
- 选择：将搜索输入框移动到顶部工具行，使用弹性布局（搜索框可伸缩，计数与按钮保留最小可见宽度）。
- 原因：满足新视觉结构要求，同时保证窄宽度下可用性。
- 备选方案：保留第二行搜索框，仅在视觉上靠近顶部行。
- 不选原因：仍保留双行断层，不符合本次布局目标。

### Decision 6: 顶部区采用 sticky 容器与滚动区分离
- 选择：将根节点标题、搜索框、计数与操作按钮放入同一 sticky top container；文件节点列表放入独立滚动容器。
- 原因：满足“根不随滚动变化”的交互预期，保证关键操作入口始终可见。
- 备选方案：整体同容器滚动，仅通过视觉弱化模拟固定。
- 不选原因：无法确保根节点与操作区持续可见，不符合需求。

## Risks / Trade-offs

- [Risk] 根节点引入后可能影响既有节点 key/path 计算，导致展开状态错乱。  
  → Mitigation: 统一 key 生成策略（root 固定 key + 相对路径 key），补充展开状态迁移单测。

- [Risk] 根节点菜单绑定 workspace path 时存在越界风险（路径拼接/传参错误）。  
  → Mitigation: 后端统一执行 workspace-bound path validation，拒绝 traversal/out-of-root。

- [Risk] 菜单复用可能引入 UI 位置/命中区冲突。  
  → Mitigation: 保持菜单触发区与折叠箭头命中区隔离，增加交互回归测试。

- [Risk] 旧测试快照基于“无根节点”结构，改动后会出现大量快照更新。  
  → Mitigation: 分层更新测试（模型、组件、e2e），明确快照变更原因并补行为断言。

- [Risk] 搜索框并入顶部行后在窄宽度下可能被按钮区挤压导致可用性下降。  
  → Mitigation: 设定输入框最小宽度与右侧按钮最小点击区，增加响应式断点测试。

- [Risk] sticky 容器与内部滚动容器组合可能引发层级/滚动同步问题。  
  → Mitigation: 明确滚动承载元素与 z-index 规范，增加滚动交互与点击命中回归测试。

## Migration Plan

1. 数据层迁移：在文件树构建阶段引入 root 节点封装，顶层条目改为 root children。
2. 交互层迁移：接入 root expand/collapse 与 context menu 触发。
3. 布局层迁移：将搜索输入框并入顶部工具行，移除独立第二行搜索容器。
4. 滚动层迁移：拆分 sticky 顶部区与列表滚动区，确保根节点顶部区固定。
5. 命令层迁移：根节点 menu action 复用目录 action dispatch，路径来源改为 workspace root。
6. 回归验证：执行文件树主链路、多 Tab、special directory progressive loading、Diff/Preview、顶部行布局与 sticky 行为回归。
7. 发布策略：灰度到开发分支，确认无回退后合并。

回滚方案：
- 保留 root wrapper feature flag（默认开）；若出现阻断回归，关闭 flag 回退到原始顶层结构。
- 回滚不涉及数据持久层 schema 变更，可快速恢复。

## Open Questions

- 根节点文案使用工作区显示名还是目录 basename（是否受 i18n/别名影响）？
- 根节点“移到废纸篓”在部分平台是否需要二次确认弹窗差异策略？
- 根节点操作入口（右侧 icon）是否与当前文件树头部按钮完全复用同一布局容器？
- 顶部工具行在极窄宽度下，搜索框是否允许折叠为 icon 弹出式输入？
- sticky 顶部区在不同平台滚动条样式下是否需要额外内边距补偿？
