## Why

当前对话输入框在 `Ctrl/Cmd + Z` 时会出现“一次撤销清空大段文本”的行为，和用户对主流编辑器“按输入节奏小步撤销”的预期不一致，导致输入安全感和编辑效率下降。该问题已经直接影响日常对话编辑体验，需要优先修复为可预测、可恢复的撤销/重做行为。

## What Changes

- 为 ChatInputBox 引入可控的 incremental undo/redo 行为，保证 `Ctrl/Cmd + Z` 与 `Ctrl/Cmd + Shift + Z`（或 `Ctrl/Cmd + Y`）按小步生效。
- 在输入框发生程序化文本更新（如补全插入、文件标签渲染、Prompt 注入）时，统一纳入同一历史事务模型，避免原生历史栈被整段覆盖。
- 定义撤销/重做的最小可感知粒度（默认 `400ms` 连续输入窗口、空白/换行/粘贴/选择替换作为事务边界），使行为接近通用文本编辑器。
- 明确程序化改写规则：仅当“逻辑文本或选区发生变化”才写入事务；纯 DOM 重排（逻辑文本不变）不得新增历史事务。
- 历史快照从“文本+单点光标”升级为“文本+选区（selection start/end）”，保障替换类编辑回放一致性。
- 补充回归测试，覆盖普通输入、补全选择、IME 场景、程序化改写后撤销链路。
- 增加跨平台快捷键兼容策略：mac 优先 `Cmd+Z / Cmd+Shift+Z`，Windows 优先 `Ctrl+Z / Ctrl+Y` 并兼容 `Ctrl+Shift+Z`。

## 目标与边界

### 目标

- 让撤销/重做在 ChatInputBox 中具备稳定、可预测的小步行为。
- 消除“单次撤销导致整段清空”的高风险体验。
- 在不改变现有发送、补全、文件标签等主流程的前提下完成能力增强。

### 边界

- 仅覆盖对话输入框（ChatInputBox）编辑行为，不扩展到文件编辑器、设置输入框等其他组件。
- 仅处理前端输入栈与键盘交互，不涉及后端存储协议变更。
- 仅在 ChatInputBox 获得焦点时接管 undo/redo 快捷键，避免对全局快捷键和其他输入组件造成副作用。
- 历史栈采用有界容量（如 100 条事务）与可控事务窗口，避免无界内存增长。
- 仅“逻辑文本或选区变化”的程序化改写参与历史；纯展示层 DOM 重排（逻辑文本不变）不得入栈。

## 非目标

- 不在本次变更中重构所有 `contenteditable` 相关架构。
- 不引入跨会话持久化的撤销历史。
- 不改变现有发送快捷键语义（Enter/Cmd+Enter）及其平台映射。

## Capabilities

### New Capabilities

- `composer-incremental-undo-redo`: 为 ChatInputBox 提供可控的小步撤销/重做能力，并约束程序化改写与用户输入共享一致的历史语义。

### Modified Capabilities

- （无）

## 技术方案对比与取舍

| 方案 | 核心思路 | 优点 | 风险/缺点 | 结论 |
|---|---|---|---|---|
| A. 继续依赖浏览器原生 contenteditable undo 栈 | 不增加自定义历史，只修补局部逻辑 | 改动最小、实现快 | 程序化 `innerText/innerHTML` 改写仍会污染历史，行为不可控，难稳定复现与修复 | 不采用 |
| B. 引入应用层 undo/redo 历史栈（推荐） | 将用户输入与程序化更新统一记录为事务，显式处理撤销/重做快捷键 | 行为可预测、测试可控、可按产品定义粒度 | 需要新增历史管理模块并梳理更新入口 | 采用 |

取舍说明：本问题本质是“原生历史与程序化改写冲突”。若不掌控历史事务边界，无法稳定保证小步撤销。为避免反复补丁，选择方案 B 作为一次性治理。

## Impact

- Affected frontend modules:
  - `src/features/composer/components/ChatInputBox/ChatInputBox.tsx`
  - `src/features/composer/components/ChatInputBox/hooks/*`（键盘处理、输入同步、标签渲染相关）
- Affected behavior:
  - 输入撤销/重做快捷键行为
  - 程序化文本更新与历史栈的交互一致性
- Dependencies/APIs:
  - 无新增外部依赖（优先内部实现）
  - 无后端 API 变更

## 验收标准

- 在连续输入普通文本后，多次 `Ctrl/Cmd + Z` 可按小步回退，不出现首步即清空整段文本。
- 连续输入在默认 `400ms` 窗口内合并为单事务；遇到空白、换行、粘贴、选择替换时形成新事务。
- 在完成补全插入（如 `@文件`、`/命令`）后，撤销先回退最近一次编辑操作，再逐步回退历史内容。
- 在 IME 输入（中文/日文/韩文）后撤销行为稳定，不出现异常跳步或整段丢失。
- `Ctrl/Cmd + Shift + Z`（或 `Ctrl/Cmd + Y`）可逐步重做，且与撤销链路对称。
- 文件标签渲染等纯 DOM 重排（逻辑文本不变）不会额外增加撤销步数。
- 在替换类编辑与多次 undo/redo 后，文本与选区位置保持一致恢复。
- 现有发送快捷键、补全菜单导航、文件标签渲染功能保持兼容，无破坏性回归。
- 在 mac 上验证 `Cmd+Z / Cmd+Shift+Z`，在 Windows 上验证 `Ctrl+Z / Ctrl+Y`（并验证 `Ctrl+Shift+Z` 兼容），行为一致且不冲突。
- 当焦点不在 ChatInputBox 时，本能力不得拦截或影响其他区域的默认快捷键行为。

## 兼容性与回归门禁

### 平台兼容矩阵（必须通过）

| 平台 | Undo | Redo | 备注 |
|---|---|---|---|
| macOS | `Cmd+Z` | `Cmd+Shift+Z` | 与系统编辑器习惯一致 |
| Windows | `Ctrl+Z` | `Ctrl+Y` / `Ctrl+Shift+Z` | 双 redo 映射均需可用 |
| Linux | `Ctrl+Z` | `Ctrl+Shift+Z` | 与 Windows 语义兼容（不强制 `Ctrl+Y`） |

### 非回归清单（必须通过）

- 发送快捷键行为不变：`Enter` / `Cmd+Enter`（或现有平台映射）无变化。
- 补全菜单导航行为不变：上下选择、确认、关闭不受 undo/redo 接管影响。
- 文件标签展示行为不变：标签渲染与高亮保持现状，且纯重排不新增撤销步数。
- 焦点边界不回归：非 ChatInputBox 焦点下，系统不得消费 undo/redo 事件。
