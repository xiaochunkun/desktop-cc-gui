# Worktree Explicit Base Selection Follow-up - Design

## Context

首版能力已归档，但仍存在 8 个未收口任务，集中在“失败体验”和“发布门禁”两个维度。  
当前风险不是“功能不可用”，而是“用户不确定发生了什么”与“上线缺乏一致校验证据”。

## Goals / Non-Goals

**Goals:**

- 统一工作树创建失败态映射与文案语义。
- 明确创建后发布结果展示与失败补救路径。
- 明确 `Gemini` 入口未开放状态，避免误触与误解。
- 完成发布门禁文档化与验证闭环。

**Non-Goals:**

- 不新增协议字段与后端能力。
- 不重做已归档的主流程交互（显式 baseRef、双栏弹窗、统一 `+`）。
- 不处理与本变更无关的 Git History 回归问题。

## Decisions

### Decision 1: 采用“错误类型 -> i18n key”单向映射表

- 原因：避免组件内散落 if/else，保证文案一致性和可维护性。
- 备选：在每个调用点本地判断错误字符串。
- 取舍：映射表更稳定，可测试性更强。

### Decision 2: 发布结果展示分离“本地创建结果”与“远端发布结果”

- 原因：首版最容易引起误解的是“本地成功但远端失败”。
- 备选：合并为单状态。
- 取舍：双状态展示更符合真实执行语义，减少恐慌误判。

### Decision 3: Gemini 入口保留但禁用，并附 `(未开放)` 标签

- 原因：入口发现性保留，同时不制造“点击无响应”体验。
- 备选：直接隐藏 Gemini。
- 取舍：禁用态更清晰，也为后续开放保留位置一致性。

### Decision 4: 发布门禁采用文档化清单 + strict validate 结果

- 原因：这 8 项中 3 项属于交付治理，必须可追溯。
- 备选：仅口头/聊天确认。
- 取舍：文档化可审计，可用于后续归档复盘。

## Risks / Trade-offs

- [Risk] 文案映射覆盖不全导致 fallback 到原始错误字符串  
  → Mitigation: 为三大高频错误（baseRef/路径冲突/分支名）补单测。

- [Risk] 发布结果展示与真实返回结构不一致  
  → Mitigation: 增加集成测试断言 tracking 与 retry command 渲染条件。

- [Risk] `Gemini` 禁用态样式在不同主题下可读性不足  
  → Mitigation: 增加浅色/深色手工验收项。

- [Risk] strict validate 受仓库历史失败项干扰  
  → Mitigation: 记录“目标 change/spec 通过”与“全量仓库失败项”分离结论。

## Migration Plan

1. 完成错误映射与结果展示代码改造。
2. 补齐 `Gemini` 禁用态及说明。
3. 执行回归测试与 strict validate（目标项）。
4. 输出发布 checklist（风险、回滚、观测点）。
5. 通过后归档 follow-up change。

回滚策略：

- 如文案映射引发误导，可回滚到原通用错误展示，不影响核心创建链路。
- 如发布结果区异常，可隐藏结果扩展区域，仅保留创建主结果。

## Open Questions

- `Gemini` 从“未开放”切换到“灰度开放”时，是否需要能力探测而非固定禁用？
- 发布 checklist 是否需要沉淀为仓库级模板（供后续变更复用）？
